# Драйвер IEC 60870-5-104 в Alpha Platform: когда протокол сложнее, чем кажется

Представьте: вас просят подключить старый РТУ на подстанции к новой диспетчерской системе. Казалось бы — IEC-104, стандартный протокол, "просто TCP с телемеханикой". Но стоило начать интеграцию, как выяснилось: асинхронный обмен с нумерацией последовательности на трёх уровнях, плюс парсинг ASDU с дескрипторами качества. И это ещё до первых граблей с таймаутами на реальном объекте.

В этой статье разбираю, как устроен модуль IEC-104 в Alpha Platform — не просто "как настроить", а что происходит под капотом и почему так важно понимать внутреннее устройство протокола для диагностики проблем на объектах.

## Анатомия протокола: почему три типа фреймов?

Первый вопрос, который возникает при изучении IEC 60870-5-104 — зачем такая сложность? Почему нельзя было просто передавать данные по TCP, как делает тот же Modbus?

Ответ кроется в том, что IEC-104 изначально проектировался для телемеханики в энергетике, где критичны два требования:
- **Надёжная доставка команд управления** (представьте, что команда отключения не дошла)
- **Синхронизация времени событий** (авария в 14:23:15.123 — точность до миллисекунд)

Поэтому поверх TCP добавили ещё один уровень — APCI (Application Protocol Control Information), который оборачивает полезную нагрузку APDU (Application Protocol Data Unit).

### Три типа фреймов — три разные задачи

```
APCI Header (6 bytes):
+--------+--------+--------+--------+--------+--------+
|  0x68  | Length |  Send  |  Recv  |  Ctrl  |  Ctrl  |
|  (1B)  |  (1B)  |  (2B)  |  (2B)  |  (1B)  |  (1B)  |
+--------+--------+--------+--------+--------+--------+
```

**I-format (Information)** — это наш основной "грузовик" для данных. Каждый I-фрейм несёт счётчики N(S) и N(R), которые отслеживают последовательность отправки и подтверждения. Зачем? Потому что TCP гарантирует доставку, но не гарантирует, что приложение обработало данные. Команда "Отключить автомат" могла дойти до TCP-буфера, но застрять в очереди обработки.

**S-format (Supervisory)** — это "расписка о получении" без данных. Когда устройство получило пачку I-фреймов и обработало их, оно отправляет S-фрейм с обновлённым N(R). Экономия трафика — не нужно дублировать APDU в подтверждениях.

**U-format (Control)** — управляющие команды протокола. Здесь живут STARTDT (запуск передачи данных), STOPDT (корректная остановка) и TESTFR (проверка связи).

### Пример реального диалога

Когда Alpha.Server подключается к РТУ, происходит примерно такой диалог:

```hex
68 04 07 00 00 00  // U-format: STARTDT act (мастер: "начинаем работу?")
68 04 0B 00 00 00  // U-format: STARTDT con (РТУ: "готов!")
```

И только после этого начинается обмен I-фреймами с данными. Если связь молчит больше t3 секунд (обычно 20), отправляется TESTFR — аналог ping'а на прикладном уровне.

## Жизненный цикл соединения: от TCP SYN до первых данных

Теперь представим полную историю соединения — от TCP SYN до первых данных телемеханики:

### 1. TCP handshake
```
Client → Server: SYN
Server → Client: SYN-ACK  
Client → Server: ACK
```
Пока стандартное TCP-соединение. Но данных ещё нет.

### 2. IEC-104 активация
```
Client → Server: STARTDT act (0x68 0x04 0x07 0x00 0x00 0x00)
Server → Client: STARTDT con (0x68 0x04 0x0B 0x00 0x00 0x00)
```

Протокол переходит из состояния **STOPPED** в **STARTED**. Важно: без STARTDT con никаких данных не будет — протокол просто проигнорирует попытки отправки I-фреймов.

### 3. Общий опрос (General Interrogation)
```
Client → Server: C_IC_NA_1 (тип 100) с причиной "активация"
Server → Client: поток M_SP_NA_1, M_ME_NA_1... с причиной "запрос"
Server → Client: C_IC_NA_1 с причиной "завершение активации"
```

Это синхронизация состояний: клиент получает снимок всех важных параметров на момент подключения.

### 4. Спонтанная передача
После GI сервер отправляет только изменения — когда сработала защита, изменилось положение коммутатора, или пришло новое измерение.

### Почему не сразу STARTED?

На одном энергообъекте мы наблюдали странную картину: соединение устанавливалось, но данных не было. Wireshark показывал корректный TCP handshake, но дальше — тишина. Оказалось, что старое РТУ требовало дополнительную паузу между TCP connect и STARTDT act — внутренняя логика РТУ не успевала инициализироваться. Добавили задержку 500мс в конфигурацию, проблема ушла.

## Настройки k и w: математика надёжности

Вот тут начинается интересное. Параметры k и w — это не просто цифры в конфигурации, это компромисс между пропускной способностью и надёжностью.

**k-parameter** (обычно 12) — максимальное количество неподтверждённых I-фреймов. Зачем ограничивать? Представьте: мастер отправил 100 команд управления, а связь оборвалась. Какие команды выполнились, какие нет? Без подтверждений это неизвестно.

**w-parameter** (обычно 8) — когда отправлять S-фрейм с подтверждением. Если w=1, то подтверждаем каждый I-фрейм — надёжно, но половина трафика уходит на служебку. Если w=k, то подтверждение только когда буфер заполнится — эффективно, но ошибка в одном фрейме заблокирует весь поток.

Почему k=12, а не 20? Потому что при разрыве связи нужно повторно отправить все неподтверждённые фреймы. 12×256 байт = 3КБ — разумный объём для буферизации. А при k=50 может не хватить памяти на старом контроллере.

### Пример расчёта пропускной способности

При k=12, w=8 и средней длине I-фрейма 20 байт:
- 8 I-фреймов = 8×26 байт (APCI+APDU) = 208 байт данных
- 1 S-фрейм = 6 байт подтверждения
- Эффективность = 208/(208+6) ≈ 97%

При w=1 эффективность падает до ~81% — каждый I-фрейм требует S-фрейм.

## Конфигурация в Alpha.DevStudio: что происходит за кнопкой "Build"

Когда вы нажимаете "Build" в Alpha.DevStudio, начинается не просто компиляция — а процесс трансформации конфигурации в runtime-структуры Alpha.Server.

### Шаг 1: Парсинг адресной карты

Alpha.DevStudio читает вашу Excel-таблицу или XML-карту адресов:

```
Внутренний тег    | IOA  | Type ID | Описание
------------------|------|---------|------------------
TEMP_REACTOR_1    | 1001 | M_ME_NA_1 | Температура реактора
PUMP_STATUS_1     | 2001 | M_SP_NA_1 | Состояние насоса  
CMD_VALVE_CLOSE   | 3001 | C_SC_NA_1 | Команда закрытия
```

Но внутри генерируется не таблица, а хеш-карта для быстрого поиска:

```c
// Генерируемая структура для runtime
struct IOAMapping {
    uint32_t internal_tag_id;
    uint16_t ioa;
    uint8_t type_id;  
    uint8_t direction; // 0=read, 1=write
} mappings[MAX_MAPPINGS];
```

### Шаг 2: Генерация конфигурации протокола

Параметры станции превращаются в C-структуру:

```c
struct Station104Config {
    uint16_t common_address;  // Номер станции в ASDU
    uint32_t t0, t1, t2, t3;  // Таймауты в миллисекундах
    uint16_t k, w;            // Параметры оконного протокола
    char remote_ip[16];       // IP адрес удалённой станции
    uint16_t remote_port;     // TCP порт (обычно 2404)
};
```

### Шаг 3: Валидация и оптимизация

DevStudio выполняет проверки, которые сэкономят время на отладке:
- Уникальность IOA в пределах станции
- Соответствие Type ID направлению (команды только на запись)
- Диапазоны адресов (IOA до 16777215 согласно стандарту)
- Корректность таймаутов (t1 < t3, t0 > t1+t2)

### Шаг 4: Развёртывание на целевой машине

После сборки конфигурация копируется на Linux-сервер Alpha.Server через SCP/SFTP. Затем сигнал SIGHUP заставляет Alpha.Server перечитать конфигурацию без остановки работающих соединений — hot reload.

Это важно для промышленных объектов: обновление конфигурации не должно прерывать сбор данных с критических датчиков.

## Буферизация: что происходит при обрыве связи

Представим ситуацию: волоконно-оптическая линия между подстанцией и диспетчерской оборвалась на 5 минут. Что происходит с данными?

### Многоуровневая архитектура буферов

Alpha.Server использует каскад буферов:

1. **Acquisition buffer** — кольцевой буфер для данных от источников (Modbus RTU, аналоговый ввод)
2. **Tag cache** — оперативный кеш текущих значений всех тегов
3. **Protocol buffer** — очередь исходящих IEC-104 сообщений  
4. **Archive buffer** — буфер для historian при недоступности БД

При обрыве IEC-104 соединения:
- Acquisition buffer продолжает получать данные от устройств
- Tag cache обновляется в реальном времени
- Protocol buffer накапливает изменения для отправки после восстановления
- Archive buffer сохраняет данные локально

### Восстановление после переподключения

Когда связь восстанавливается, происходит не "дамп всех буферов", а интеллектуальная синхронизация:

1. **General Interrogation** — клиент запрашивает текущие состояния всех объектов
2. **Differential update** — сервер отправляет только изменения с момента разрыва  
3. **Archive sync** — накопленные исторические данные передаются в background режиме

Почему не отправлять все накопленные данные сразу? Потому что после 5-минутного разрыва может накопиться 10000+ изменений. Лавина данных заблокирует канал и может привести к новому таймауту.

### Пример реальной оптимизации

На одном водоканале мы столкнулись с проблемой: после кратковременных обрывов связи (~30 сек) SCADA получала огромные пачки данных, которые не успевала обрабатывать. Анализ показал, что быстроменяющиеся аналоговые сигналы (давление в трубопроводе) генерировали сотни событий за время разрыва.

Решение: добавили в конфигурацию "deadband" фильтр — изменения менее 0.5% от диапазона не передаются. Трафик снизился в 10 раз при сохранении информативности данных.

## Грабли: истории из траншей

### История первая: загадка исчезающих подтверждений

**Симптомы**: Соединение разрывалось ровно каждые 3 минуты 20 секунд, несмотря на активный обмен данными.

**Расследование**: Wireshark показал интересную картину — мастер отправлял I-фреймы с корректными N(S), но slave периодически "забывал" отправлять S-фреймы с подтверждением. Когда счётчик неподтверждённых фреймов достигал k=12, мастер закрывал соединение по таймауту t1.

**Корень проблемы**: Старое РТУ использовало кооперативную многозадачность, и при большой нагрузке обработка протокольного стека откладывалась. За это время накапливалось более k фреймов.

**Решение**: Снизили k с 12 до 8, увеличили t1 с 15 до 25 секунд. Дополнительно настроили приоритет процесса IEC-104 на максимум в конфигурации РТУ.

### История вторая: часовые пояса и хаос времени

**Симптомы**: События в SCADA отображались с timestamp на 3 часа раньше реального времени, но только зимой. Летом время было корректным.

**Расследование**: Команды синхронизации времени C_CS_NA_1 содержали локальное время московской зоны, но РТУ интерпретировало их как UTC. Зимой разница UTC+3, летом UTC+4 из-за перехода на летнее время.

```c
// НЕПРАВИЛЬНО: отправка локального времени
time_t local_time = time(NULL);
struct tm* tm_info = localtime(&local_time);
send_time_sync(tm_info);

// ПРАВИЛЬНО: явное указание UTC и смещения
time_t utc_time = time(NULL);
struct tm* utc_tm = gmtime(&utc_time);
int utc_offset = get_local_utc_offset(); // +3 или +4 часа
send_time_sync_with_offset(utc_tm, utc_offset);
```

**Решение**: Модифицировали драйвер для отправки UTC времени с явным указанием часового пояса в дополнительном информационном элементе.

### История третья: firewall против keepalive

**Симптомы**: Соединение стабильно работало на тестовом стенде, но на объекте разрывалось через ровно 30 минут бездействия в ночные часы.

**Расследование**: Тестовый стенд находился в локальной сети, а на объекте между SCADA и РТУ был enterprise firewall с NAT timeout = 1800 секунд. В ночные часы события редки, t3=20 секунд оказывалось недостаточно — TESTFR команды терялись в NAT translation table.

**Решение**: Активировали TCP keepalive на уровне socket'а с параметрами keepidle=10s, keepintvl=3s, keepcnt=3:

```c
int keepalive = 1;
setsockopt(sock, SOL_SOCKET, SO_KEEPALIVE, &keepalive, sizeof(keepalive));

int keepidle = 10;   // Первый keepalive через 10с простоя
int keepintvl = 3;   // Повтор каждые 3с  
int keepcnt = 3;     // Разрыв после 3х неудач
setsockopt(sock, IPPROTO_TCP, TCP_KEEPIDLE, &keepidle, sizeof(keepidle));
setsockopt(sock, IPPROTO_TCP, TCP_KEEPINTVL, &keepintvl, sizeof(keepintvl));
setsockopt(sock, IPPROTO_TCP, TCP_KEEPCNT, &keepcnt, sizeof(keepcnt));
```

TCP keepalive работает на транспортном уровне и поддерживает NAT mapping даже при отсутствии прикладного трафика.

### История четвёртая: дескрипторы качества как источник проблем

**Симптомы**: В historian записывались явно некорректные значения: температура реактора -273°C, давление в трубопроводе 999999 бар.

**Расследование**: Драйвер игнорировал Quality Descriptors в информационных объектах и записывал в БД всё, что приходило по сети.

```c
// НЕПРАВИЛЬНО: слепая запись без проверки качества
if (asdu->type == M_ME_NA_1) {
    float value = normalized_to_float(info_obj->value);
    historian_write(ioa, value, timestamp); // Записали -273°C!
}

// ПРАВИЛЬНО: анализ дескрипторов качества  
if (asdu->type == M_ME_NA_1) {
    uint8_t qds = info_obj->quality;
    if (qds & QDS_IV) {
        // Бит IV (Invalid) — значение недостоверно
        historian_write_invalid(ioa, timestamp);
        log_warning("IOA %d: значение помечено как недостоверное", ioa);
    } else if (qds & QDS_BL) {
        // Бит BL (Blocked) — измерение заблокировано
        historian_write_blocked(ioa, timestamp);  
    } else {
        // Значение корректно, можно записывать
        float value = normalized_to_float(info_obj->value);
        historian_write(ioa, value, timestamp);
    }
}
```

**Корень проблемы**: При калибровке датчиков их значения временно становились недостоверными, но РТУ продолжало их передавать с установленным битом IV. Игнорирование QDS приводило к записи мусора в архив.

## Alpha Platform: архитектурные решения

### Нативная интеграция без OPC-прослоек

Основное отличие Alpha.Server от многих решений — драйвер IEC-104 встроен в ядро платформы, а не реализован как отдельный OPC-сервер с последующим мостом. Это даёт несколько преимуществ:

1. **Единая модель данных** — теги IEC-104 живут в том же адресном пространстве, что и Modbus, OPC UA, внутренние переменные
2. **Отсутствие двойной буферизации** — данные не копируются между процессами через OPC DA/UA
3. **Прямой доступ к TCP-стеку** — возможность fine-tuning socket параметров (Nagle algorithm, buffer sizes)

### Архитектура горячего резерва

Для критически важных объектов Alpha.Server поддерживает активный резерв:

```
Primary Alpha.Server ──┐
                       ├── Heartbeat ──→ IEC-104 Clients  
Secondary Alpha.Server ─┘     (VIP)
        │
        └─── Shared Storage (данные, конфигурации)
```

Переключение происходит на основе heartbeat через виртуальный IP адрес. Время переключения при отказе primary сервера — менее 10 секунд (время обнаружения + TCP timeout на клиентской стороне).

**Почему не кластер с балансировкой?** IEC-104 — это stateful протокол с нумерацией последовательности. Нельзя просто перенаправить TCP-соединение на другой сервер — придётся разрывать все активные сессии и переустанавливать их заново.

### DMZ и инверсия соединений

Для промышленных сетей с жёсткой сегментацией Alpha.Server поддерживает reverse connection pattern:

```
DMZ Zone          Internal Network
┌─────────────┐   ┌──────────────┐
│ Alpha.Server │←──│ Alpha.Server │
│ (Gateway)    │   │ (Main SCADA) │  
└─────────────┘   └──────────────┘
      ↑                  ↑
  IEC-104 out      Внутренняя шина данных
  (порт 2404)
```

Alpha.Server в DMZ работает как преобразователь протоколов — принимает исходящие соединения от внешних систем, но получает данные из внутренней сети через собственную шину Alpha.AccessPoint.

Это решает проблему firewall правил: нет необходимости открывать порты из DMZ во внутреннюю сеть.

## Выбор протокола: когда IEC-104, а когда нет?

После 15 лет работы с промышленными протоколами у меня сложилось понимание их "экологических ниш":

### Modbus RTU/TCP: простота прежде всего
- **Объём**: до 1000 тегов
- **Применение**: локальные контуры, простые механизмы
- **Плюсы**: universal поддержка, отладка обычным telnet'ом
- **Минусы**: нет временных меток, примитивная диагностика ошибок

### IEC 60870-5-104: золотая середина телемеханики  
- **Объём**: 1000-50000 тегов
- **Применение**: подстанции, энергообъекты, водоканал
- **Плюсы**: временные метки, дескрипторы качества, стандартизация
- **Минусы**: сложность отладки, legacy совместимость

### OPC UA: будущее промышленности
- **Объём**: 50000+ тегов  
- **Применение**: современные производства, Industry 4.0
- **Плюсы**: security, semantic interoperability, pub/sub
- **Минусы**: высокая сложность, ресурсоёмкость

Примерное время внедрения типового проекта:
- Modbus: 1-2 недели
- IEC-104: 1-2 месяца (с отладкой)
- OPC UA: 3-6 месяцев (с разработкой information model)

## Заключение: протокол живёт в деталях

IEC 60870-5-104 остаётся основой телемеханики в энергетике не благодаря простоте — а благодаря продуманному балансу между функциональностью и complexity. Каждая особенность протокола (k/w параметры, дескрипторы качества, логика переходов между состояниями) решает реальную инженерную задачу.

Alpha Platform предоставляет готовое решение с нативной Linux поддержкой, но понимание протокольных нюансов остаётся критически важным. В промышленной автоматизации "просто работает" случается редко — обычно нужно разбираться с legacy оборудованием, сетевой инфраструктурой и edge case'ами, которые не описаны в стандартах.

Следующая статья в серии будет посвящена OPC UA информационным моделям в Alpha Platform — как перейти от "soup of tags" к семантически структурированным данным.
