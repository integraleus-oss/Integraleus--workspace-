# Перенос проектов Alpha.DevStudio с Windows на Astra Linux: что реально ждёт на практике

**Тег:** Гайды · **Время чтения:** 12 мин · **Автор:** Автоматчик

Задача звучала просто: взять проект Alpha.DevStudio 4.1, разработанный на Windows 10, и запустить на Astra Linux. На деле — три дня отладки, десяток неочевидных проблем и несколько уроков, которые мы теперь фиксируем, чтобы следующая миграция прошла за полдня.

---

## Зачем вообще переносить

Импортозамещение — не абстрактная инициатива, а конкретные требования заказчиков. На нескольких проектах мы столкнулись с ситуацией: АСУ ТП работает на Альфа платформе, проект разрабатывался под Windows, а серверы на площадке — Astra Linux. Alpha.DevStudio 4.1 поддерживает обе ОС (а ещё Ubuntu, Альт и РЕД ОС), так что принципиальных препятствий нет. Но дьявол, как обычно, в деталях.

Эта статья — про те самые детали: структура проекта, кодировки, пути, DLL и типичные грабли, на которые мы наступили сами.

---

## Что внутри проекта DevStudio

Прежде чем переносить, полезно понимать, что именно мы переносим. Проект Alpha.DevStudio — это набор файлов и папок:

- **Файл решения** — контейнер, содержит ссылки на входящие проекты
- **Файлы проектов** — конфигурация каждого модуля
- **Папки конфигураций** — скомпилированные модули (.binom)
- **Исходники** — процедуры Alpha.Om, формулы, HMI-схемы (.om, .xml)
- **Ресурсы** — изображения (.png, .svg), шрифты (.ttf), библиотеки (.dll)

Типичная структура: папки Components (объекты), Scripts (Alpha.Om), HMI (мнемосхемы), Resources (внешние файлы). Всё это нужно перенести целиком, ничего не потеряв и не сломав ссылки.

## Три главных различия между Windows и Linux

### Пути к файлам

На Windows мы привыкли к:

```
C:\Alpha\Projects\MyProject\Components\Tank01.xml
```

На Linux это превращается в:

```
/opt/alpha/projects/MyProject/Components/Tank01.xml
```

Обратные слеши → прямые, буква диска исчезает. DevStudio умеет адаптировать разделители автоматически при открытии проекта. Но если в конфигурации где-то зашит абсолютный путь вроде `D:\MyProject\Images\valve.png` — Linux его не поймёт. Мы нашли такие ссылки в трёх местах из пяти проектов, которые переносили.

**Решение:** перед миграцией пройдитесь по конфигурации и замените абсолютные пути на относительные. Это можно сделать ещё на Windows, и проект продолжит работать.

### Регистр символов

На Windows `Tank01.xml`, `tank01.xml` и `TANK01.XML` — один и тот же файл. На Linux — три разных файла.

На одном из проектов у нас в конфигурации была ссылка `valve.png`, а файл на диске назывался `Valve.PNG`. На Windows всё работало, на Linux — мнемосхема не нашла изображение. Причём ошибка не всегда очевидна: компиляция проходит, а картинка просто не отображается в HMI.

**Решение:** перед переносом проведите аудит имён файлов. Проще всего привести все имена к нижнему регистру и обновить ссылки. На Linux это можно проверить командой:

```bash
# Поиск файлов с заглавными буквами в расширениях
find /opt/alpha/projects/MyProject -name "*.PNG" -o -name "*.BMP" -o -name "*.XML"
```

### Кодировки: CP1251 vs UTF-8

Самая коварная проблема. На Windows текстовые файлы нередко оказываются в кодировке CP1251 (она же Windows-1251). На Linux стандарт — UTF-8. Если в скриптах Alpha.Om или в комментариях XML есть русские символы в CP1251, на Linux они превращаются в кракозябры.

Признаки проблемы:
- Нечитаемые комментарии в скриптах
- Ошибки компиляции на именах объектов с кириллицей
- Искажённый текст на мнемосхемах HMI

На практике мы столкнулись с этим на проекте, где имена сигналов содержали русские названия (например, «Давление_входное»). На Windows компилировалось, на Linux — ошибка.

**Решение:** конвертировать файлы в UTF-8 до переноса. Проверить кодировку можно утилитой `file`:

```bash
file -i Scripts/*.om
# Если видите charset=iso-8859-1 или charset=unknown-8bit — скорее всего CP1251
```

Для конвертации:

```bash
iconv -f CP1251 -t UTF-8 script.om > script_utf8.om
mv script_utf8.om script.om
```

## Пошаговая миграция: как мы это делаем

За несколько проектов у нас выработался алгоритм из пяти этапов.

### Этап 1. Аудит на исходной системе

Открываем проект в DevStudio на Windows и фиксируем:

- Список внешних зависимостей (вкладки Dependencies и References в свойствах проекта)
- Все файлы с русскими символами — проверяем кодировку
- Абсолютные пути в конфигурации — заменяем на относительные
- Внешние DLL-библиотеки — составляем список, ищем Linux-аналоги (.so)

Этот этап занимает от часа до полного рабочего дня, в зависимости от размера проекта. Но он экономит дни на следующих этапах.

### Этап 2. Подготовка архива

- Закрываем DevStudio (убеждаемся, что файлы не залочены)
- Копируем папку проекта целиком, включая подпапки
- Если используются кастомные библиотеки — экспортируем отдельно
- Фиксируем версии всех модулей

Мы используем Git для версионирования проектов DevStudio (поддерживается наравне с SVN). Это сильно упрощает жизнь: можно откатиться, сравнить изменения, вести параллельные ветки под Windows и Linux.

### Этап 3. Перенос файлов

Варианты доставки:
- **SCP/SFTP** — для сетевого переноса (наш основной способ)
- **USB** — для изолированных систем без сети
- **tar-архив** — универсальный вариант

```bash
# Копирование по SCP
scp -r user@windows-host:/C/Alpha/Projects/MyProject /opt/alpha/projects/

# Или через архив
tar -xzf myproject.tar.gz -C /opt/alpha/projects/

# Права доступа
chmod -R 755 /opt/alpha/projects/MyProject
chown -R alpha:alpha /opt/alpha/projects/MyProject
```

### Этап 4. Адаптация на Linux

Здесь основная работа:

1. **Кодировки** — конвертируем CP1251 → UTF-8 (если не сделали на этапе 1)
2. **Пути** — DevStudio адаптирует разделители автоматически, но абсолютные пути нужно исправить вручную
3. **Регистр файлов** — проверяем соответствие между именами файлов и ссылками
4. **DLL → .so** — заменяем Windows-библиотеки на Linux-аналоги (подробнее ниже)
5. **Шрифты** — устанавливаем в систему, если HMI использует нестандартные

```bash
# Установка шрифтов в Astra Linux
sudo cp *.ttf /usr/share/fonts/truetype/
sudo fc-cache -fv
fc-list | grep "НужныйШрифт"
```

### Этап 5. Тестирование

Чек-лист, который мы проходим на каждом проекте:

1. **Компиляция** — проект собирается без ошибок
2. **Загрузка конфигурации** — Alpha.Server принимает проект
3. **HMI** — мнемосхемы открываются, русский текст отображается, изображения на месте
4. **Скрипты Alpha.Om** — процедуры выполняются корректно
5. **Связь с оборудованием** — драйверы подключаются к контроллерам
6. **Архивирование** — исторические данные пишутся
7. **Тревоги** — аварийные ситуации обрабатываются

Для автоматизации проверок используем CLI DevStudio (`devstudio.cli.exe` с командами `compile`, `build`, `deploy`, `deploy-status`). Это позволяет встроить проверку в CI/CD.

## Проблема DLL → .so: самое сложное

Windows DLL не работают в Linux. Точка. Если проект использует внешние .dll для интеграции с оборудованием или нестандартных вычислений, нужна замена.

Что мы делаем на практике:

1. **Ищем Linux-версии от вендора** — многие производители оборудования уже предоставляют кроссплатформенные SDK
2. **Перепроектируем на стандартные протоколы** — заменяем проприетарные DLL на связку через Modbus TCP или OPC UA. Это вообще хорошая практика, даже без миграции
3. **Wine** — теоретически возможно, но для продакшена мы не рекомендуем. Непредсказуемое поведение, проблемы с производительностью
4. **Контейнеризация** — в крайнем случае, изолируем Windows-компонент в виртуальной машине

На одном проекте замена проприетарной DLL на OPC UA заняла два дня, но в итоге система стала надёжнее: стандартный протокол вместо закрытой библиотеки, мониторинг подключения из коробки.

## Типичные ошибки, на которые мы наступили

**«Модуль не найден»** — чаще всего проблема путей. Проверяем ссылки в конфигурации проекта, наличие модулей в указанных директориях. На Linux путь `/opt/Alpha/` и `/opt/alpha/` — разные вещи.

**«Ошибка кодировки в имени объекта»** — кириллица в CP1251. Конвертируем в UTF-8, перекомпилируем.

**Изображения не отображаются в HMI** — регистр имён файлов. На Windows `Logo.PNG` находился по ссылке `logo.png`, на Linux — нет.

**Медленная загрузка проекта** — проверяем права доступа и расположение файлов. Сетевые папки (NFS/SMB) работают заметно медленнее локального диска. Антивирус на Linux тоже может сканировать файлы проекта в реальном времени.

```bash
# Правильные права для файлов проекта
find /opt/alpha/projects/MyProject -type f -exec chmod 644 {} \;
find /opt/alpha/projects/MyProject -type d -exec chmod 755 {} \;
```

## Требования к окружению на Linux

Alpha.DevStudio 4.1 на Linux требует:
- **.NET 8.0 Runtime** — обязательно
- **libgdiplus** — для графических компонентов
- **SVN** — если проект версионируется через Subversion (Git работает из коробки)

Поддерживаемые ОС: Astra Linux, Ubuntu, Альт, РЕД ОС. На каждой из них мы запускали проекты — принципиальных различий между дистрибутивами нет, но на Astra Linux нужно учитывать особенности мандатного контроля доступа.

## Мониторинг после переноса

Миграция не заканчивается успешной компиляцией. На первом проекте мы считали, что раз собралось и запустилось — значит, готово. Через неделю выяснилось, что один из скриптов Alpha.Om работал на 30% медленнее из-за некорректной работы с временными файлами (путь был зашит с обратными слешами, и система каждый раз создавала новый файл вместо перезаписи).

Мы рекомендуем после миграции сравнить ключевые метрики:
- **Время загрузки проекта** — должно быть сопоставимо с Windows
- **Скорость обновления HMI** — частота перерисовки мнемосхем
- **Использование CPU и памяти** — резкий рост указывает на проблему
- **Пропускная способность I/O** — количество обрабатываемых сигналов в секунду

Alpha.Diagnostics 2.2 помогает собрать эти данные. Мы обычно оставляем мониторинг на первые две недели после переноса.

## Когда миграция не нужна

Честность требует сказать: не всегда перенос оправдан.

- **Критическая зависимость от Windows DLL**, для которых нет Linux-аналогов и замена нецелесообразна — проще оставить на Windows или вынести в отдельный сервис
- **Устаревший проект**, который проще пересоздать с нуля на целевой платформе
- **Система с коротким жизненным циклом** — если через год планируется полная замена, тратить время на миграцию нет смысла

## Что мы вынесли из практики

Несколько принципов, к которым мы пришли после нескольких миграций:

- **Начинайте с аудита, не с копирования.** Час аудита экономит день отладки
- **Используйте Git с первого дня.** Версионирование проектов DevStudio через Git (или SVN) — страховка при любых проблемах
- **Пишите в UTF-8 изначально.** Если проект ещё на этапе разработки — сразу настройте UTF-8. Это устранит целый класс проблем
- **Избегайте абсолютных путей.** Относительные пути работают на обеих платформах
- **Тестируйте на Linux регулярно, а не один раз в конце.** Мы запускаем компиляцию на Linux через CLI (`devstudio.cli.exe compile`) в CI после каждого коммита
- **Для новых проектов сразу ведите кроссплатформенную разработку.** Без проприетарных DLL, без кириллицы в CP1251, без абсолютных путей — и переезд будет безболезненным
