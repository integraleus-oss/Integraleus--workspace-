# Драйвер IEC 60870-5-104 в Alpha Platform: архитектура модуля и подводные камни реальных объектов

Вас просят подключить старое РТУ на подстанции к новой диспетчерской. IEC-104, стандартный протокол — «просто TCP с телемеханикой». Но стоило начать интеграцию: асинхронный обмен с нумерацией последовательности, парсинг ASDU с шестью независимыми флагами качества, и пять таймаутов, каждый из которых при неверной настройке роняет соединение. Это ещё до первых граблей с NAT и часовыми поясами на реальном объекте.

Разбираю, как устроен модуль IEC-104 Master в Alpha.Server — не пересказ документации, а архитектурные решения и реальные проблемы, которые стандарт не описывает.

## Зачем три типа фреймов поверх TCP

Первый вопрос при чтении ГОСТ Р МЭК 870-5-104-2004: зачем усложнять? TCP уже гарантирует доставку — Modbus TCP обходится без дополнительных обёрток.

Ответ в двух требованиях телемеханики энергетики:

- **Подтверждение обработки, а не доставки** — TCP гарантирует, что байты дошли до буфера. Не гарантирует, что приложение их обработало. Команда «Отключить автомат» могла сесть в TCP-буфер и застрять в очереди обработки.
- **Миллисекундные метки времени событий** — авария в 14:23:15.123 должна быть зафиксирована именно с этой меткой, а не с моментом получения.

Поэтому поверх TCP добавлен уровень APCI — Application Protocol Control Information, обёртка для APDU:

```
APCI (6 байт):
+--------+--------+--------+--------+--------+--------+
|  0x68  | Length | Ctrl 1 | Ctrl 2 | Ctrl 3 | Ctrl 4 |
|  Start |  APDU  | ─────────── Control Field ──────── |
+--------+--------+--------+--------+--------+--------+
```

Шесть байт. Start byte (всегда 0x68), длина APDU, четыре байта управляющего поля. Интерпретация управляющего поля зависит от типа фрейма — и здесь начинается архитектура:

**I-format** — информационный фрейм. Четыре байта Control Field кодируют два 15-битных счётчика: N(S) — номер отправленного фрейма и N(R) — номер последнего подтверждённого принятого. Младший бит Ctrl 1 = 0 — признак I-формата. Это «грузовик» для данных: ASDU с измерениями, событиями, командами.

**S-format** — подтверждение без данных. Ctrl 1 = 0x01. Передаёт только N(R) — «я обработал всё до фрейма N». Экономия: не нужно дублировать APDU в подтверждениях.

**U-format** — управление протоколом. Ctrl 1 = 0x03. Шесть функций: STARTDT act/con (запуск передачи), STOPDT act/con (остановка), TESTFR act/con (проверка связи).

## Жизненный цикл соединения

### 1. TCP handshake

Стандартный SYN → SYN-ACK → ACK. Данных ещё нет.

### 2. Активация канала данных

```
Мастер → Slave: STARTDT act  (0x68 0x04 0x07 0x00 0x00 0x00)
Slave → Мастер: STARTDT con  (0x68 0x04 0x0B 0x00 0x00 0x00)
```

Протокол переходит из состояния STOPPED в STARTED. Критично: без STARTDT con попытки отправки I-фреймов игнорируются. Протокол молчит, Wireshark показывает только TCP ACK, инженер думает что «связь есть, а данных нет».

### 3. Общий опрос (General Interrogation)

```
Мастер → Slave: C_IC_NA_1 (тип 100), причина передачи = активация
Slave → Мастер: поток M_SP_NA_1, M_ME_NA_1... причина = ответ на запрос
Slave → Мастер: C_IC_NA_1, причина = завершение активации
```

Синхронизация состояний: мастер получает снимок всех параметров на момент подключения. В Alpha.Server период GI настраивается в конфигурации модуля — для критических объектов имеет смысл опрашивать каждые 15–30 минут, для стабильных — раз в час.

### 4. Событийный обмен

После GI slave отправляет только спонтанные изменения — сработала защита, изменилось положение коммутатора, обновилось измерение. Модуль IEC-104 Master в Alpha.Server — **событийный**: данные приходят при изменении значения параметра, а не по циклическому опросу. Это принципиально отличает архитектуру от Modbus, где мастер вынужден постоянно опрашивать регистры.

## Параметры k, w и таймауты: компромиссы

### Оконный протокол

**k** (типичное значение 12) — максимальное количество неподтверждённых I-фреймов. Аналог TCP window, но на прикладном уровне. Зачем ограничивать? При разрыве связи все неподтверждённые фреймы — под вопросом. При k=12 и среднем фрейме ~25 байт это 300 байт для буферизации. При k=50 старый контроллер с 64 КБ RAM может не справиться.

**w** (типичное значение 8) — порог отправки S-фрейма с подтверждением. При получении w I-фреймов модуль отправляет подтверждение с текущим N(R). Компромисс:

| w | Поведение | Эффективность |
|---|-----------|---------------|
| 1 | Подтверждение каждого фрейма | ~81% (каждый 26-байтный I-фрейм + 6-байтный S-фрейм) |
| 8 | Подтверждение пачки из 8 | ~97% (208 байт данных + 6 байт подтверждения) |
| k | Подтверждение только при заполнении буфера | Максимальная, но ошибка блокирует весь поток |

Рекомендация: w ≤ 2/3 × k. При k=12 ставим w=8 — разумный баланс.

### Пять таймаутов

| Параметр | Типичное значение | Назначение |
|----------|-------------------|------------|
| t0 | 30 с | Таймаут TCP-соединения |
| t1 | 15 с | Таймаут подтверждения отправленных данных. Нет ACK за t1 — выставляем COMM_FAILURE |
| t2 | 10 с | Интервал отправки подтверждения принятых данных. Истёк t2 — модуль высылает S-фрейм с числом принятых пакетов |
| t3 | 20 с | Интервал тестирования связи. Нет трафика за t3 — отправляем TESTFR |

Обязательное соотношение: **t2 < t1** — сначала подтверждение, потом таймаут. Нарушение этого условия — одна из самых частых ошибок конфигурации, которую Alpha.DevStudio должен валидировать на этапе сборки проекта.

## Архитектура модуля в Alpha.Server

### Событийная модель

Модуль IEC-104 Master — событийный компонент ядра Alpha.Server. Не отдельный процесс, не OPC-прослойка — модуль работает внутри сервера и взаимодействует с ядром через подписки на изменения сигналов:

1. **Инициализация**: чтение конфигурации → подписка у ядра на уведомления об изменениях сигналов → проверка адресов → построение очереди опроса
2. **Установка соединений**: подключение к станциям и их источникам → отправка STARTDT act → ожидание STARTDT con → GI
3. **Рабочий режим**: получение спонтанных данных → запись в ОБД (оперативную базу данных) через интерфейсы ядра → уведомление других модулей (OPC DA Server, Historian, Logics Module)

Если адрес пришедшего сигнала не обслуживается модулем — данные игнорируются. Это не баг, а архитектурное решение: модуль обрабатывает только подписанные сигналы, остальные отбрасываются на уровне ядра.

### Масштабирование: станции, источники, каналы

Иерархия адресации:

```
Модуль IEC-104 Master
├── Станция 1 (common address = номер КП)
│   ├── Источник 1 (активный)
│   │   ├── Канал 1 (TCP, основной)
│   │   ├── Канал 2 (TCP, резервный)
│   │   ├── Канал 3
│   │   └── Канал 4
│   ├── Источник 2 (резервный)
│   │   └── ...
│   ├── Источник 3
│   └── Источник 4
├── Станция 2
│   └── ...
└── ... (до 65 534 станций)
```

- До **65 534** подчинённых станций (двухбайтная адресация, 65535 = широковещательный)
- До **4 источников** на станцию (для резервирования оборудования)
- До **4 каналов связи** на источник (для резервирования линий)
- **Параллельный опрос** активного источника по всем каналам одновременно. При параллельном опросе используются типы данных с меткой времени — чтобы корректно упорядочить события, пришедшие по разным каналам.

### Режимы РАБОТА и РЕЗЕРВ

Модуль работает в двух режимах, переключаемых ядром:

**РАБОТА** — полный функционал: опрос, приём данных, выдача команд ТУ и ТР.

**РЕЗЕРВ** — два варианта (настраивается в конфигурации):
- *Опрос без команд*: модуль продолжает получать данные, но команды управления заблокированы. Минимальное время переключения в РАБОТА — данные актуальны, сессии открыты.
- *Закрытие соединений*: модуль отключается от станций. Экономия ресурсов, но переключение в РАБОТА требует полной инициализации.

Для критических объектов первый вариант — единственный разумный. При отказе primary-сервера резервный уже имеет актуальные данные и переключается за секунды, а не за минуту переподключения.

### Качество сигналов: шесть флагов

Одна из сильных сторон IEC-104 по сравнению с Modbus — развитая система дескрипторов качества. Каждый информационный объект несёт шесть независимых битовых флагов:

| Флаг | Назначение | Типичная причина |
|------|-----------|-----------------|
| **IV** (Invalid) | Значение недостоверно | Датчик отключен, калибровка |
| **NT** (Not Topical) | Значение неактуально | Давно не обновлялось |
| **SB** (Substituted) | Значение подставлено | Ручной ввод оператором |
| **BL** (Blocked) | Передача заблокирована | Локальная блокировка на КП |
| **OV** (Overflow) | Переполнение диапазона | Выход за пределы шкалы |
| **CY** (Carry) | Перенос счётчика | Переполнение интегратора |

Модуль IEC-104 Master в Alpha.Server транслирует эти флаги в собственную систему качества ядра:

| Качество Alpha.Server | Числовое значение | Смысл |
|-----------------------|-------------------|-------|
| GOOD | 192 | Значение достоверно |
| UNCERTAIN | 64 | Связь есть, значение ещё не получено |
| COMM_FAILURE | 24 | Потеря связи со станцией |
| CONFIG_ERROR | 4 | Ошибка конфигурации сигнала |
| NOT_CONNECTED | 8 | Устройство не инициализировано |
| DEVICE_FAILURE | 12 | Ошибка инициализации |
| LOCAL_OVERRIDE | 216 | Достоверно, введено вручную |

Промежуточные состояния — UNCERTAIN и NOT_CONNECTED — часто упускают при интеграции. Инженер видит «данные есть» и принимает 0.0 за реальное значение, хотя качество = 64 означает «сигнал подписан, но значение ещё ни разу не пришло».

## Грабли: четыре истории с реальных объектов

### Исчезающие подтверждения

**Симптомы**: соединение разрывается ровно каждые 3 минуты 20 секунд, хотя обмен данными активен.

**Расследование**: Wireshark показывает — мастер отправляет I-фреймы с корректными N(S), но slave периодически «забывает» отправить S-фрейм. Счётчик неподтверждённых фреймов доходит до k=12, мастер фиксирует таймаут t1 и закрывает соединение.

**Корень проблемы**: старое РТУ с кооперативной многозадачностью. Под нагрузкой обработка протокольного стека откладывается — накапливается более k фреймов без подтверждения.

**Решение**: снизили k с 12 до 8, увеличили t1 с 15 до 25 секунд. Дополнительно повысили приоритет процесса IEC-104 на РТУ.

**Урок**: параметры k и w — не «просто настройки». Они должны учитывать реальную производительность конечного оборудования, а не только стандарт.

### Часовые пояса и хаос времени

**Симптомы**: события в SCADA отображаются со сдвигом на 3 часа, но только зимой. Летом время корректное.

**Расследование**: команды синхронизации C_CS_NA_1 содержали локальное время московского часового пояса. РТУ интерпретировало его как UTC. Зимой разница UTC+3, летом (когда в России ещё был переход) — UTC+4.

```c
// Неправильно: отправка локального времени
time_t local = time(NULL);
struct tm *tm_info = localtime(&local);
send_time_sync(tm_info);

// Правильно: UTC с явным указанием смещения
time_t utc = time(NULL);
struct tm *utc_tm = gmtime(&utc);
int offset_hours = get_local_utc_offset();
send_time_sync_with_tz(utc_tm, offset_hours);
```

**Урок**: стандарт ГОСТ Р МЭК 870-5-104 описывает формат метки времени CP56Time2a, но оставляет интерпретацию часовых поясов на реализацию. Всегда договариваться: UTC или локальное, и фиксировать это в проектной документации.

### Firewall убивает ночные сессии

**Симптомы**: на тестовом стенде всё стабильно. На объекте — разрыв ровно через 30 минут бездействия, только ночью.

**Расследование**: между SCADA и РТУ стоит enterprise firewall с NAT timeout = 1800 секунд. Ночью событий мало. t3=20 секунд — TESTFR отправляется, но ответ теряется в протухшей NAT-трансляции.

**Решение**: активировали TCP keepalive на уровне сокета:

```c
int keepalive = 1;
setsockopt(sock, SOL_SOCKET, SO_KEEPALIVE, &keepalive, sizeof(keepalive));

int keepidle = 10;    // Первый keepalive через 10 с простоя
int keepintvl = 3;    // Повтор каждые 3 с
int keepcnt = 3;      // Разрыв после 3 неудач
setsockopt(sock, IPPROTO_TCP, TCP_KEEPIDLE, &keepidle, sizeof(keepidle));
setsockopt(sock, IPPROTO_TCP, TCP_KEEPINTVL, &keepintvl, sizeof(keepintvl));
setsockopt(sock, IPPROTO_TCP, TCP_KEEPCNT, &keepcnt, sizeof(keepcnt));
```

TCP keepalive работает на транспортном уровне и поддерживает NAT mapping даже при отсутствии прикладного трафика IEC-104.

**Урок**: t3 и TESTFR решают проблему на прикладном уровне. Но если между мастером и slave стоит stateful firewall с NAT — нужен keepalive на уровне TCP. Два разных механизма для двух разных задач.

### Мусор в historian из-за игнорирования флагов качества

**Симптомы**: в архиве — температура реактора –273 °C, давление 999 999 бар.

**Расследование**: драйвер записывал в БД всё, что приходило по сети, игнорируя Quality Descriptors:

```c
// Неправильно: слепая запись
if (asdu->type == M_ME_NA_1) {
    float value = normalized_to_float(info_obj->value);
    historian_write(ioa, value, timestamp);  // Записали –273 °C!
}

// Правильно: проверка флагов качества
if (asdu->type == M_ME_NA_1) {
    uint8_t qds = info_obj->quality;
    if (qds & QDS_IV) {
        // Invalid — значение недостоверно
        historian_write_with_quality(ioa, NAN, timestamp, QUALITY_BAD);
    } else if (qds & QDS_BL) {
        // Blocked — заблокировано на КП
        historian_write_with_quality(ioa, last_good_value, timestamp, QUALITY_UNCERTAIN);
    } else if (qds & QDS_NT) {
        // Not Topical — давно не обновлялось
        historian_write_with_quality(ioa, value, timestamp, QUALITY_UNCERTAIN);
    } else {
        float value = normalized_to_float(info_obj->value);
        historian_write_with_quality(ioa, value, timestamp, QUALITY_GOOD);
    }
}
```

**Корень проблемы**: при калибровке датчиков значения временно становятся недостоверными. РТУ продолжает их передавать с установленным битом IV. Система Alpha.Server правильно транслирует флаги IEC-104 в свои качества (GOOD/UNCERTAIN/COMM_FAILURE), но если данные передаются через OPC в стороннюю систему — проверку качества должен выполнять каждый получатель.

## Буферизация при обрыве связи

Волоконно-оптическая линия оборвалась на 5 минут. Что происходит?

Модуль IEC-104 Master обнаруживает потерю связи по таймауту t1. Сигналам выставляется качество COMM_FAILURE. Далее — таймаут потери связи (настраивается в конфигурации). В течение этого периода модуль пытается восстановить соединение. Если связь не восстановлена — плохое качество для всех сигналов.

После восстановления:

1. **Переустановка сессии**: TCP connect → STARTDT → STARTDT con
2. **General Interrogation**: полный снимок текущего состояния всех объектов
3. **Спонтанные данные**: slave накопил события за время разрыва и начинает отправлять их с метками времени

Проблема: если за 5 минут на slave накопилось 10 000+ событий, лавина данных может заблокировать канал. Стандарт не регламентирует приоритизацию — это ответственность реализации.

Практический приём: настройка deadband для быстроменяющихся аналогов. Изменения менее 0.5% от диапазона не генерируют событие. На водоканале это снизило объём данных после переподключения в 10 раз при сохранении информативности.

В Alpha.Server при переполнении очереди данных (размер настраивается) опрос станции не останавливается, но модуль фиксирует событие в журнале. Данные, не поместившиеся в очередь, отбрасываются — осознанный выбор: лучше потерять часть истории, чем заблокировать приём актуальных данных.

## Нативная интеграция vs OPC-прослойки

Модуль IEC-104 встроен в ядро Alpha.Server. Не отдельный OPC-сервер с мостом, а компонент единой архитектуры. Что это даёт:

**Единое адресное пространство** — теги IEC-104 живут рядом с Modbus, OPC UA, IEC 61850, S7, FINS, BACnet и ещё двумя десятками протоколов. Логический модуль ядра может оперировать любыми сигналами независимо от источника.

**Без двойной буферизации** — данные из сетевого стека идут напрямую в ОБД ядра. Нет промежуточного процесса, нет копирования через COM/DCOM (OPC DA) или TCP (OPC UA).

**Единая система качества** — флаги IEC-104 (IV/BL/NT/SB/OV) транслируются в качества ядра (GOOD/UNCERTAIN/COMM_FAILURE) на этапе приёма. Все потребители — HMI, historian, алармы, логика — работают с одной системой качества.

**Горячее резервирование** — ядро управляет переключением РАБОТА/РЕЗЕРВ для всех модулей единообразно. Резервный Alpha.Server в режиме опроса держит открытые сессии IEC-104 и актуальные данные — переключение за секунды.

| Применение | Локальные контуры, простые механизмы | Подстанции, энергообъекты, водоканал | Современные производства, Industry 4.0 |
| Метки времени | Нет | Да (CP56Time2a, мс) | Да (DateTime, 100 нс) |
| Качество данных | Нет | 6 флагов (IV/BL/SB/NT/OV/CY) | StatusCode + SourceTimestamp |
| Безопасность | Нет | Нет (IEC 62351 — опционально) | Встроенная (сертификаты, шифрование) |
| Типичное время внедрения | 1–2 недели | 1–2 месяца (с отладкой) | 3–6 месяцев (с information model) |

IEC-104 занимает нишу между простотой Modbus и мощью OPC UA. Для энергетики, водоканала, нефтегаза с удалёнными объектами — это стандарт де-факто. Alpha.Server поддерживает все три протокола в обе стороны (Master/Slave, Client/Server), что позволяет строить шлюзы между мирами: IEC-104 от КП на подстанции → Alpha.Server → OPC UA к MES.

## Заключение

IEC 60870-5-104 остаётся основой телемеханики не благодаря простоте, а благодаря продуманному балансу. Каждая «сложность» протокола — k/w параметры, пять таймаутов, шесть флагов качества, логика STARTDT/STOPDT — решает конкретную инженерную задачу.

Модуль IEC-104 Master в Alpha.Server реализует событийную архитектуру с параллельным опросом по нескольким каналам, поддержкой до 65 534 станций и горячим резервированием. Но понимание протокольных нюансов — соотношение таймаутов, поведение NAT, интерпретация часовых поясов — остаётся задачей инженера. В промышленной автоматизации «просто работает» — это результат правильной конфигурации, а не магии.
