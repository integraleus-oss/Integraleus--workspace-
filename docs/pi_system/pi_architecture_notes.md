## 1. Основы PI System

### 1.1 Что такое PI System
- **Определение**: Инфраструктура данных для сбора, хранения и управления данными процессов
- **Основные компоненты**:
  - Источники данных (DCS, PLC, SCADA, LIMS и др.)
  - Сборщики данных (PI Interface, PI Connector, AVEVA Adapter)
  - PI Data Archive + PI Asset Framework Server (PI Server)
  - Инструменты визуализации (PI Vision, PI ProcessBook)

### 1.2 Архитектура PI System
- **Простая система**: источник → интерфейс → PI Server → клиент
- **PI Server = PI Data Archive + PI Asset Framework**
- **Новая терминология** (с 2018):
  - PI Data Archive — компонент хранения временных рядов
  - PI Server — общий термин для Data Archive + AF

### 1.3 Временной синтаксис PI
**Фиксированное время**: `5-jan-92 12:34`, `25-sep-12`

**Относительное время**:
- `*` — сейчас
- `t` — сегодня (00:00)
- `y` — вчера (00:00)
- `fri` — последняя пятница
- `may` — текущий день в мае

**Смещения времени**:
- `*-1h` — час назад
- `t+8h` — 8:00 сегодня
- `-1d` — день назад от текущего времени

**Единицы времени**: s, m, h, d, w, mo, y

## 2. Планирование PI System

### 2.1 Высокая доступность (HA)
- **PI Data Archive HA**: репликация между серверами
- **PI Collective Manager**: управление кластером PI серверов
- **N-Way Buffering**: буферизация на несколько серверов
- **Interface Failover**: автоматическое переключение между интерфейсами

### 2.2 Безопасность
**Рекомендуемый подход**: Windows Integrated Security

**Аутентификация**:
- PI Mappings (Windows User → PI Identity)
- PI Trusts (для старых API)
- Group Managed Service Accounts (gMSA) — рекомендуется

**gMSA преимущества**:
- Автоматическое управление паролями
- Централизованное хранение
- Без доступа пользователей к паролям
- Требует PowerShell для настройки

### 2.3 Новые возможности: OpenID Connect (OIDC)
- Claims-based аутентификация
- Поддержка корпоративных Identity Providers
- TLS шифрование
- Снижение зависимости от Active Directory

## 3. Установка и настройка

### 3.1 Требования к оборудованию
**Диск**:
- PI Data Archive: 11 MB на 1000 точек + 1 MB
- Архивы: 10 MB на 1000 точек
- Event Queue: 5 MB на 1000 точек

**Память**: 
- Рекомендуется размер архива = 2-3 KB × количество точек
- Файловый кеш Windows может использовать всю RAM на 64-bit

### 3.2 Размеры архивов по количеству точек
| Лицензированные точки | Размер архива |
|----------------------|--------------|
| 0 - 87,381          | 256 MB       |
| 87,382 - 174,762    | 512 MB       |
| 174,763 - 349,525   | 1024 MB      |
| 349,526 - 699,050   | 2048 MB      |
| 699,051+            | 4096+ MB     |

### 3.3 Поддерживаемые ОС
**Продакшен**:
- Windows Server 2022 (рекомендуется)
- Windows Server 2019
- Windows Server 2016

**Тестирование**: Windows 10

## 4. PI Interfaces

### 4.1 Функции PI Interface
1. Подключение к источнику данных
2. Установка временных меток
3. Форматирование данных
4. Отправка в PI Data Archive

### 4.2 Основные параметры
- `/HOST` — адрес PI сервера
- `/ID` — идентификатор экземпляра интерфейса
- `/PS` — Point Source (источник точек)
- `/F` — частота сканирования (scan class)

**Пример scan class**:
- `/f=5,10` — каждые 5 секунд со смещением 10 секунд
- `/f=00:01:00,00:00:05` — каждую минуту со смещением 5 секунд

### 4.3 Архитектурные паттерны
**Тип A**: Источник данных + PI Interface на одной машине  
**Тип B**: Источник, Interface и PI Data Archive на разных машинах  
**Тип C**: PI Interface на сервере PI Data Archive

### 4.4 Безопасность интерфейсов
- **С PI API для WIS**: Windows Mapping
- **Со старым PI API**: PI Trust
- **Формат Trust**: IP адрес + имя приложения (например, OPCpE)

## 5. PI Points

### 5.1 Типы данных
- **Digital**: дискретные состояния
- **Float32/Float64**: числа с плавающей точкой
- **Int16/Int32**: целые числа
- **String**: текстовые данные

### 5.2 Ключевые атрибуты точек
- **Archiving**: включение архивирования (1/0)
- **Compressing**: сжатие (1/0)
- **CompDev**: отклонение для сжатия
- **CompMax**: максимальное время сжатия (по умолчанию 8 часов)
- **ExcDev**: отклонение исключения
- **ExcMax**: максимальное время исключения (по умолчанию 10 минут)
- **Span/Zero**: диапазон значений
- **PointSource**: должен совпадать с /PS в интерфейсе
- **Location1**: обычно ID экземпляра интерфейса
- **Location4**: номер scan class
- **InstrumentTag**: имя точки в источнике данных

### 5.3 Future Data
- **Время хранения**: от января 1970 до января 2038
- **Stored Values = 1**: разрешение будущих данных
- **Отклонение будущих событий**: >10 минут для обычных точек
- **Отдельные архивы**: future data никогда не переходит в исторические архивы

## 6. Data Flow и обработка данных

### 6.1 Reporting by Exception
- **Цель**: уменьшение шума в данных
- **Алгоритм**: dead band (зона нечувствительности)
- **Параметры**: ExcDev (отклонение), ExcMax (максимальное время)

### 6.2 Compression
- **Цель**: хранение минимума данных для точной реконструкции
- **Алгоритм**: анализ наклона данных
- **Параметры**: CompDev, CompMax

### 6.3 Рекомендации по настройке
- **CompDev**: минимальное измеримое изменение прибора
- **ExcDev**: половина от CompDev
- **Отключение**: ExcDev = 0, ExcMax = 0, Compressing = Off

### 6.4 Архивы PI
- **Исторические архивы**: фиксированный размер, последовательные по времени
- **Future архивы**: динамический размер, начальный размер 1 MB
- **Период future архивов**: 1 календарный месяц
- **Требование**: все архивы должны быть зарегистрированы

## 7. PI Buffer Subsystem

### 7.1 Назначение
- **Буферизация данных** при потере связи с PI сервером
- **N-Way Buffering**: отправка данных на несколько серверов
- **Надежность**: предотвращение потери данных

### 7.2 Конфигурация
- **Приоритет запуска**: PI Buffer Subsystem должен стартовать до PI Interface
- **Зависимости сервисов**: PIBuffss → PI Interface services
- **gMSA поддержка**: с версии 4.9.0.37

## 8. PI Connectors

### 8.1 Основные типы
**PI Connector (1st Gen)**:
- Автоматическое создание точек и AF элементов
- Собственная буферизация
- Поддержка failover

**PI Connector (2nd Gen)**:
- Работает с PI Connector Relay
- Управление через PI Data Collection Manager
- НЕТ поддержки failover

**AVEVA Adapter**:
- Отправка в OMF endpoints
- PI Web API и CONNECT data services

### 8.2 Различия с PI Interface
- Автоматическое создание метаданных
- Без exception reporting
- Без timestamping на стороне коннектора

## 9. Высокая доступность (HA)

### 9.1 Компоненты HA
- **PI Data Archive Replication**: синхронизация конфигурации
- **PI Collective Manager**: управление кластером
- **N-Way Buffering**: буферизация на все члены коллектива
- **Interface Failover**: переключение между интерфейсами

### 9.2 Настройка HA
1. **Формирование Collective**: PI Collective Manager
2. **Тестирование N-Way Buffering**
3. **Подготовка второго узла сбора данных**
4. **Реализация Failover**
5. **Итоговое тестирование**

## 11. Практические рекомендации

### 11.1 Best Practices
- Использование gMSA для сервисных аккаунтов
- Настройка proper exception и compression параметров
- Регулярное обновление DST правил (localhost.tz)
- Синхронизация времени всех компонентов
- Использование фиксированных IP для серверов

### 11.2 Частые проблемы
- **Разница времени >10 минут**: отклонение future events
- **Неправильные PI Mappings/Trusts**: проблемы подключения
- **Неверные атрибуты точек**: отсутствие данных
- **Неправильный порядок запуска сервисов**

### 11.3 Инструменты диагностики
- **PI SMT**: основной инструмент администрирования
- **apisnap**: проверка PI API соединения
- **PI SDK Utility**: проверка PI SDK соединения
- **Network Manager Statistics**: мониторинг подключений
- **Message Logs**: диагностика ошибок
