# Альфа платформа vs AVEVA PI System — Детальное сравнение (v2)

> Дата: 2026-04-01
> Источники:
> - Альфа платформа: документация 6500+ стр., конспекты 50 модулей
> - AVEVA PI System: 4 учебных курса (~27 000 строк), docs.aveva.com, aveva.com

---

## 1. Позиционирование — ключевое отличие

| | Альфа платформа | AVEVA PI System |
|---|---|---|
| **Разработчик** | АО «Атомик Софт» (Россия, бренд Automiq) | AVEVA / Schneider Electric (Великобритания) |
| **Что это** | Полноценная SCADA-платформа: сбор + управление + визуализация + архив | Платформа данных: сбор + архив + контекстуализация + аналитика. НЕ SCADA |
| **Оператор может** | Управлять процессом (команды на ПЛК, квитирование, ввод уставок) | Только наблюдать (мониторинг, тренды, анализ) |
| **Реестр Минцифры** | ✅ №13866 | ❌ |
| **Санкционные риски** | Нет | Иностранное ПО, ограничения поставок |

> **Главный вывод:** Это продукты разных классов. Альфа — SCADA, из которой оператор управляет процессом. PI System — инфраструктура данных, которая подключается к существующим SCADA/DCS и делает данные доступными для анализа. В крупных проектах PI System часто работает РЯДОМ со SCADA, а не вместо неё.

## 2. Архитектура

### Альфа платформа
```
ПЛК/Датчики → Alpha.Server (сбор + логика + команды)
                  ↓
              Alpha.Historian (архив)
                  ↓
              Alpha.HMI (мнемосхемы, управление)
              Alpha.HMI.Alarms (тревоги)
              Alpha.Reports (отчёты)
```
Всё в одной платформе, единый конфиг через Alpha.DevStudio.

### AVEVA PI System
```
SCADA/DCS/PLC → PI Interface / PI Connector / AVEVA Adapter
                     ↓
                PI Data Archive (хранение временных рядов)
                     ↓
                PI Asset Framework (модель активов, SQL Server)
                     ↓
                PI Analysis Service (расчёты)
                PI Vision (веб-визуализация)
                PI DataLink (Excel)
                PI Integrator (BI)
```
PI Server = PI Data Archive + PI Asset Framework (терминология с 2018).
Каждый компонент — отдельный сервис со своими портами, настройками, лицензией.

### Сравнение подходов

| | Альфа | PI System |
|---|---|---|
| **Установка** | Один инсталлятор, одна конфигурация | Множество компонентов: DA, AF, SQL Server, Analysis Service, Vision, интерфейсы — каждый отдельно |
| **Администрирование** | Alpha.DevStudio + CLI | PI SMT, PSE, PI Builder, PI Collective Manager, PI ICU — разные утилиты для разных задач |
| **Кривая обучения** | Один продукт — освоил и работаешь | Экосистема из 10+ компонентов, каждый со своей логикой |
| **Гибкость** | Монолит — всё или ничего | Можно взять только Data Archive, без AF и Vision |

### Три поколения сборщиков PI System
1. **PI Interface** — классика, 450+ протоколов, exception reporting на стороне интерфейса, требует ручного создания точек, поддержка failover
2. **PI Connector (1st/2nd Gen)** — автоматическое создание точек и AF-структуры, НО 2nd Gen не поддерживает failover
3. **AVEVA Adapter** — кроссплатформенный (Windows/Linux/Docker), OMF-протокол, отправка в PI Web API или CONNECT

### Буферизация
| | Альфа | PI System |
|---|---|---|
| **При потере связи** | Данные буферизуются в Alpha.Server | PI Buffer Subsystem (PIBufSS) — отдельный сервис |
| **N-Way Buffering** | ❌ | ✅ Буферизация на несколько серверов одновременно |
| **Disconnected Startup** | ✅ Alpha.Server работает автономно | ✅ PI Interface пишет конфиг на диск и стартует без DA |

## 5. Модель активов (контекстуализация)

| | Альфа | PI System |
|---|---|---|
| **Модель** | Дерево сигналов в DevStudio + Alpha.Domain | PI Asset Framework (AF) — полноценная семантическая модель |
| **Масштаб** | Зависит от проекта | До 100+ миллионов атрибутов |
| **Иерархии** | Одно дерево | Множественные иерархии (по географии, процессу, оборудованию) |
| **Шаблоны** | Пользовательские типы в HMI (аналог ООП) | AF Templates с наследованием, автоматическим обновлением всех экземпляров |
| **Подстановочные параметры** | ❌ | ✅ Substitution Parameters (`\|Element\|`, `\|@Asset\|`) |
| **Связи между активами** | Через адресацию сигналов | Явные references + Category-based группировка |
| **Внешние данные** | ✅ SQLConnector (в сигналы) + подключение DLL/SO | ✅ Table Lookup (из SQL прямо в атрибуты AF) |
| **Единицы измерения** | ✅ | ✅ UOM с автоматической конвертацией между единицами |
| **Безопасность на уровне объектов** | ✅ Alpha.Security | ✅ По элементам, категориям, шаблонам (Read/ReadWrite/Admin) |
| **Хранилище модели** | Конфигурация проекта | Microsoft SQL Server |

### AF — ключевая сила PI System
Asset Framework — это не просто «дерево тегов». Это:
- Семантическая модель предприятия (Завод → Цех → Линия → Насос → Подшипник)
- Шаблоны: создал «Насос» один раз с 20 атрибутами → применил к 500 насосам → изменил шаблон → все 500 обновились
- Rollup: «средняя температура по всем насосам цеха» — одной настройкой
- Кросс-сервер: один AF-элемент может ссылаться на данные из разных PI Data Archive
- Рекомендуемая глубина: до 7-8 уровней, до 1000 дочерних элементов

## 6. Аналитика и вычисления

| | Альфа | PI System |
|---|---|---|

| **Шаблонизация** | ❌ Каждый сигнал отдельно | ✅ Один расчёт в шаблоне → ко всем экземплярам |
| **Backfill** | ❌ | ✅ Пересчёт по историческим данным при изменении формулы |
| **Rollup** | ❌ | ✅ Sum/Avg/Min/Max/Count по дочерним элементам |
| **Event Frame Generation** | ❌ | ✅ Автоматическое создание событий по условиям |
| **Триггеры расчётов** | По изменению сигнала | Periodic / Periodic+DataChange / Event-based |

### Встроенные функции PI Analysis
- `TagAvg()`, `TagMax()`, `TagMin()`, `TagTot()` — статистика за период
- `TagVal()` — значение в момент времени
- `NoOutput()` — подавление вывода
- `If/Then/Else` — условия
- `FindEq()`, `FindGT()` — поиск значений
- `Abs()`, `Sqrt()`, `Log()`, `Exp()` — математика
- Substitution: `'|Element|.Temperature'` — относительные ссылки

## 7. Event Frames — уникальная фича PI System

| | Альфа | PI System |
|---|---|---|
| **Фиксация событий** | Журнал тревог (Alpha.HMI.Alarms) | Event Frames — полноценные именованные периоды |
| **Шаблоны событий** | ❌ | ✅ Event Frame Templates с атрибутами |
| **Вложенность** | ❌ | ✅ Вложенные EF (Смена → Партия 1 → Фаза загрузки) |
| **Сравнение событий** | ❌ | ✅ Compare Events в PI Vision (best batch vs worst batch) |
| **Генерация по условию** | Тревоги по уставкам | ✅ EF Generation Analysis (condition-based, periodic) |
| **Output Expressions** | ❌ | ✅ Расчёты внутри EF (например, макс. температура за партию) |

Event Frames — то, чего у Альфы нет аналога. Это инструмент для ретроспективного анализа: «почему партия №47 была хуже партии №46?» — сравнение всех параметров за два временных окна.

## 18. Итоговые плюсы и минусы

### Альфа платформа — плюсы
1. ✅ **Российский продукт** — реестр Минцифры, нет санкционных рисков
2. ✅ **Полноценная SCADA** — оператор управляет процессом (команды, квитирование, уставки)
3. ✅ **Linux** — Astra, РЕД ОС, Альт, Ubuntu (критично для КИИ)
4. ✅ **Цена** — значительно дешевле PI System
5. ✅ **Бессрочная лицензия** — нет подписки
6. ✅ **Простота** — один продукт vs экосистема из 10+ компонентов

## 19. Когда что выбирать

### Выбирай Альфу, если:
- Нужна SCADA для управления процессом
- Российское предприятие, импортозамещение, КИИ
- Linux обязателен
- Бюджет ограничен
- Нужна простота (один продукт, одна IDE)

### Выбирай PI System, если:
- Нужен enterprise-историан для аналитики (не для управления)
- Нужна модель активов и цифровые двойники
- Нужна интеграция с BI/ML
- Мультисайтовая инфраструктура с облаком
- Нет ограничений по импортозамещению

### Они работают вместе:
В крупных проектах PI System подключается к Альфе (через OPC UA/DA) и забирает данные для аналитики. Альфа управляет процессом, PI System делает данные доступными для инженеров, аналитиков и руководства. Это разные слои одной архитектуры.
