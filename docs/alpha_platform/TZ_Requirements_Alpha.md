# Требования к программному обеспечению SCADA-платформы

## 1. Общие требования

### 1.1. Соответствие нормативным требованиям
ПО должно быть включено в Единый реестр российских программ для электронных вычислительных машин и баз данных (в соответствии с Приказом Минкомсвязи России №486).

### 1.2. Поддержка операционных систем
Серверные и исполняющие компоненты ПО (серверы ввода-вывода, серверы архивирования, подсистемы безопасности, веб-серверы визуализации) должны функционировать нативно под операционными системами семейства Windows (Windows 10/11, Windows Server 2019/2022) и Linux (Astra Linux SE, РЕД ОС, ALT Linux) без использования сторонних слоёв совместимости и эмуляторов (Wine, Mono и пр.). Должна обеспечиваться функциональная эквивалентность проекта автоматизации при развёртывании на различных поддерживаемых ОС.

## 2. Требования к серверу данных

### 2.1. Производительность
Сервер ввода-вывода должен обеспечивать обработку не менее 1 000 000 (одного миллиона) тегов на один экземпляр сервера.

### 2.2. Масштабируемость
ПО должно поддерживать работу не менее 4 (четырёх) серверных узлов в рамках одного проекта автоматизации с возможностью построения территориально распределённых систем.

### 2.3. Оперативная база данных реального времени
ПО должно включать собственную высокопроизводительную базу данных реального времени (БДРВ) для хранения и обработки оперативных значений технологических параметров.

### 2.4. Резервирование
ПО должно обеспечивать горячее резервирование серверных компонентов с автоматическим переключением на резервный сервер и репликацией данных между основным и резервным серверами.

## 3. Требования к коммуникационным возможностям

### 3.1. Встроенные протоколы сбора данных
Сервер данных должен обеспечивать сбор данных по следующим протоколам и спецификациям встроенными средствами платформы:

| № | Протокол / спецификация | Режим |
|---|------------------------|-------|
| 1 | ГОСТ Р МЭК 60870-5-104 | Master и Slave |
| 2 | ГОСТ Р МЭК 60870-5-101 | Master и Slave |
| 3 | ГОСТ Р МЭК 61850 | Client |
| 4 | Modbus TCP | Master и Slave |
| 5 | Modbus RTU | Master и Slave |
| 6 | OPC DA | Client и Server |
| 7 | OPC UA | Client и Server |
| 8 | OPC HDA | Client и Server |
| 9 | OPC AE | Server |
| 10 | BACnet | Client |
| 11 | MQTT | Client |
| 12 | SNMP | Manager |

### 3.2. Двусторонний обмен данными
ПО должно обеспечивать как приём (Master/Client), так и передачу (Slave/Server) данных по протоколам ГОСТ Р МЭК 60870-5-104, ГОСТ Р МЭК 60870-5-101 встроенными средствами платформы.

### 3.3. Внутренний протокол с буферизацией
ПО должно обеспечивать буферизацию данных при передаче от сервера данных к серверу архивирования. При разрыве связи между серверными компонентами данные должны сохраняться в локальном буфере и автоматически передаваться при восстановлении соединения, исключая потерю технологической информации.

### 3.4. Поддержка демилитаризованных зон (ДМЗ)
ПО должно обеспечивать возможность создания демилитаризованных зон (ДМЗ) на уровне архитектуры проекта, в том числе за счёт размещения промежуточных экземпляров серверов, выполняющих функцию конверторов протоколов и шлюзов данных между изолированными сетевыми сегментами. Должна поддерживаться инверсия инициации соединения для прохождения через межсетевые экраны.

## 4. Требования к архивированию данных

### 4.1. Встроенная система архивирования
ПО должно включать собственный компонент архивирования (сервер архивирования) для сбора и долгосрочного хранения технологических данных без обязательного использования сторонних систем управления базами данных (MS SQL Server, PostgreSQL и пр.).

## 5. Требования к визуализации и клиентской части

### 5.1. Веб-клиент
ПО должно обеспечивать работу веб-клиентов через стандартный веб-браузер без установки дополнительных плагинов, ActiveX-компонентов или иных специализированных клиентских надстроек.

## 6. Требования к эксплуатации и обслуживанию

### 6.1. Версионность конфигураций и поэтапное обновление
ПО должно поддерживать версионность конфигураций с возможностью отката к предыдущей стабильной версии. При наличии горячего резервирования должна обеспечиваться возможность поэтапного обновления конфигурации без полной остановки системы.

### 6.2. Непрерывность работы при отсутствии лицензионного ключа
ПО должно обеспечивать сохранение полнофункционального режима работы системы в случае выхода из строя или временного отсутствия аппаратного/программного лицензионного ключа защиты. Данное требование обусловлено необходимостью обеспечения непрерывности технологического процесса.

| 2.2 | ≥4 серверных узлов | Распределённые системы диспетчеризации |
| 2.3 | Собственная БДРВ | Производительность, независимость от сторонних СУБД |

| 6.1 | Версионность и поэтапное обновление | Минимизация простоев, откат при ошибках |
