# Nielsen Norman Group — UX-справочник

Источник: nngroup.com (Jakob Nielsen, Don Norman)
Использовать при работе над сайтом, калькулятором, формами, статьями.

---

## 10 эвристик юзабилити Нильсена

### 1. Видимость состояния системы
Дизайн должен **всегда** информировать пользователя, что происходит — через уместную обратную связь в разумное время.
→ Прогресс-бары, индикаторы загрузки, статусы отправки форм.

### 2. Соответствие системы реальному миру
Говори на **языке пользователя** — слова, фразы и понятия, знакомые ему, а не внутренний жаргон. Информация в естественном и логичном порядке.
→ Для АСУ-шников: "тег", "контроллер", "ОРС-сервер" — ок. "Инстанс обработчика данных" — нет.

### 3. Свобода и контроль пользователя
Чётко обозначенный "аварийный выход" из любого действия. Отмена, шаг назад.
→ Кнопка "Назад" в калькуляторе, возможность очистить форму.

### 4. Единообразие и стандарты
Пользователь не должен гадать, означают ли разные слова/ситуации одно и то же. Следовать платформенным и отраслевым конвенциям.
→ Если "Каталог" в навбаре — везде "Каталог", не "Продукты" на одной странице и "Решения" на другой.

### 5. Предотвращение ошибок
Лучше предотвратить ошибку, чем показать сообщение. Два типа: **промахи** (невнимательность) и **заблуждения** (неверная ментальная модель).
→ Валидация в реальном времени, подтверждение перед необратимыми действиями.

### 6. Узнавание вместо запоминания
Минимизировать нагрузку на память. Элементы, действия, опции должны быть видимы.
→ Лейблы форм всегда видны, не заменять их плейсхолдерами.

### 7. Гибкость и эффективность
Шорткаты для опытных пользователей. Возможность настроить частые действия.
→ Калькулятор: запомнить последний расчёт, быстрый пересчёт.

### 8. Эстетика и минимализм
Интерфейс не должен содержать нерелевантную информацию. Каждый лишний элемент конкурирует с важным.
→ Убирать "шум" со страниц. Фокус на главном действии.

### 9. Помочь распознать, диагностировать и исправить ошибки
Сообщения об ошибках: **простой язык** (без кодов), **точное описание проблемы**, **конструктивное решение**.
→ Не "Error 500", а "Не удалось отправить заявку. Проверьте подключение и попробуйте ещё раз."

### 10. Справка и документация
В идеале система работает без доп. объяснений. Но если нужна помощь — она должна быть легко находимой, краткой, с конкретными шагами.

## 3 C's микрокопии (NN/g, 2026)

### 1. Clarity (Ясность) — ПРИОРИТЕТ №1
- Пользователь знает, **где он** (навигация)
- Пользователь знает, **куда идти дальше** (информационный запах)
- Релевантность — зачем это ему
- Подходящая терминология — объяснять жаргон, если аудитория новая

### 2. Concision (Краткость) — ПРИОРИТЕТ №2
- Каждое слово должно служить цели
- Микрокопия < 3 предложений по определению
- Учитывать ограничения: заголовки < 60 символов (SERP), описания < 160
- Можно добавить пару слов ради ясности, но не больше

### 3. Character (Характер) — ПРИОРИТЕТ №3
- Тон, юмор, всё что делает текст человечным и запоминающимся
- Адаптировать под аудиторию
- **Не в ущерб ясности и краткости**

## Сообщения об ошибках (NN/g Guidelines)

### Видимость
- Показывать **рядом с источником** ошибки (близость)
- Красный, жирный, высокий контраст + иконка (не только цвет — 350М дальтоников)
- Тяжесть ошибки → стиль сообщения (тост vs модальное окно)
- **Не показывать ошибку преждевременно** — дать заполнить, потом валидировать

### Коммуникация
- **Человеческий язык** — без кодов и жаргона
- **Точно и кратко** описать проблему — не "Произошла ошибка"
- **Конструктивный совет** — что делать дальше
- **Позитивный тон, не обвинять** — не "Неверный email", а "Проверьте формат email"
- Осторожно с юмором (повторные ошибки — шутка надоедает)

### Эффективность
- Сохранять введённые данные при ошибке
- Минимизировать шаги для исправления
- Для 404: помочь найти нужное (поиск, ссылки)

## Плейсхолдеры в формах (NN/g)

### Главное правило: плейсхолдер ≠ замена лейбла

**7 причин почему плейсхолдеры вредны как замена лейблов:**
1. Исчезающий текст нагружает кратковременную память
2. Нельзя проверить заполнение перед отправкой
3. При ошибке непонятно, что исправлять
4. Мешает навигации клавиатурой (Tab)
5. Заполненное поле менее заметно (eye-tracking)
6. Путают с автозаполнением
7. Иногда приходится удалять вручную

### Правильный подход:
- **Лейбл** — всегда снаружи поля, всегда видим
- **Плейсхолдер** — только дополнительная подсказка (формат, пример)
- Важные инструкции — **вне поля**, чтобы были видны всегда
- Floating labels — компромисс, но не идеал

## Применение

### Любые формы и калькуляторы
- [ ] Лейблы полей — всегда видимы, не только плейсхолдеры
- [ ] Валидация — в реальном времени, рядом с полем
- [ ] Ошибки — конкретные ("Укажите количество тегов" вместо "Заполните поле")
- [ ] Успех — чёткое подтверждение + что дальше
- [ ] Прогресс — если многошаговый, показать где пользователь
- [ ] CTA — конкретный глагол ("Отправить заявку", не "Отправить")
- [ ] Минимум полей, каждое поле объяснено

### Навигация (сайты, веб-приложения)
- [ ] Единообразие терминов
- [ ] Хлебные крошки на вложенных страницах
- [ ] 404 — полезная страница с поиском и ссылками

### Промышленные интерфейсы (SCADA, HMI, DCS, MES)
- [ ] Алармы: что → где → серьёзность → что делать
- [ ] Подтверждение критичных действий (открытие/закрытие арматуры, пуск/стоп)
- [ ] Контекстные подсказки рядом с параметрами
- [ ] Пустые состояния: "Нет активных аварий" вместо пустого списка

### Общие тексты (везде)
- [ ] 3 C's: ясность → краткость → характер
- [ ] Язык пользователя, не маркетинговый буллшит
- [ ] Снимать возражения до их появления
