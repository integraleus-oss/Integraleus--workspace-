## 1. IEC 61508 — общий стандарт

### Назначение
IEC 61508 «Функциональная безопасность электрических, электронных и программируемых электронных систем, связанных с безопасностью» — базовый международный стандарт, определяющий требования к системам безопасности на протяжении всего жизненного цикла.

### Структура стандарта (7 частей)
- **Часть 1:** Общие требования
- **Часть 2:** Требования к E/E/PE системам
- **Часть 3:** Требования к программному обеспечению
- **Часть 4:** Определения и сокращения
- **Часть 5:** Примеры методов определения SIL
- **Часть 6:** Руководство по применению частей 2 и 3
- **Часть 7:** Обзор методов и мер

### Жизненный цикл безопасности

```
1. Определение концепции
        ↓
2. Определение области применения
        ↓
3. Анализ опасностей и рисков (HAZOP, LOPA)
        ↓
4. Определение требований безопасности
        ↓
5. Распределение требований (SIL allocation)
        ↓
6. Планирование эксплуатации и обслуживания
        ↓
┌───────────────────────────────────────┐
│   V-модель разработки                │
│                                       │
│   Требования ←────────→ Валидация     │
│       ↓                    ↑          │
│   Проектирование ←──→ Интеграция     │
│       ↓                    ↑          │
│   Реализация ←────→ Тестирование     │
│       ↓                    ↑          │
│   Модульное тестирование              │
└───────────────────────────────────────┘
        ↓
7. Ввод в эксплуатацию
        ↓
8. Эксплуатация и обслуживание
        ↓
9. Модификация / ретрофит
        ↓
10. Вывод из эксплуатации
```

### V-модель разработки ПО (IEC 61508-3)

Левая ветвь (декомпозиция):
1. Спецификация требований к ПО безопасности
2. Архитектура ПО безопасности
3. Детальное проектирование модулей
4. Кодирование

Правая ветвь (интеграция и верификация):
1. Модульное тестирование (→ верификация кода)
2. Интеграционное тестирование (→ верификация архитектуры)
3. Валидация ПО (→ верификация требований)

Каждый уровень левой ветви имеет соответствие на правой ветви.

### Ключевые концепции
- **Функция безопасности (Safety Function)** — функция, реализуемая SIS для достижения или поддержания безопасного состояния
- **Безопасное состояние** — состояние процесса, при котором отсутствует неприемлемый риск
- **Отказ опасный (Dangerous failure)** — отказ, препятствующий выполнению функции безопасности
- **Отказ безопасный (Safe failure)** — отказ, приводящий к ложному срабатыванию (переход в безопасное состояние)
- **SFF (Safe Failure Fraction)** — доля безопасных отказов = (λSD + λSU + λDD) / (λSD + λSU + λDD + λDU)

## 2. IEC 61511 — процессные производства

### Назначение
Применение IEC 61508 в отрасли непрерывных процессов (нефтегаз, химия, нефтепереработка, фармацевтика).

### Ключевые понятия

- **SIS (Safety Instrumented System)** — приборная система безопасности
- **SIF (Safety Instrumented Function)** — функция безопасности, реализуемая SIS
- **SRS (Safety Requirements Specification)** — спецификация требований безопасности

### Структура SIS

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Датчики    │ ──→ │  Логический  │ ──→ │ Исполнительные│
│   (Sensor)   │     │  решатель    │     │  элементы     │
│              │     │  (Logic      │     │ (Final Element)│
│ Давление,    │     │   Solver)    │     │ Клапаны,      │
│ Температура, │     │              │     │ отсечные      │
│ Уровень,     │     │ ПЛК безоп-   │     │ задвижки      │
│ Расход       │     │ ности (SIL)  │     │               │
└──────────────┘     └──────────────┘     └──────────────┘
```

### Жизненный цикл по IEC 61511
1. Анализ опасностей и рисков (HAZOP)
2. Распределение SIL (LOPA — Layer of Protection Analysis)
3. Разработка SRS
4. Проектирование и инжиниринг SIS
5. Монтаж, ввод в эксплуатацию
6. Валидация безопасности
7. Эксплуатация и обслуживание
8. Модификация
9. Вывод из эксплуатации

## 3. Уровни полноты безопасности SIL

### Определение
SIL (Safety Integrity Level) — дискретный уровень, определяющий требуемую надёжность функции безопасности.

### Требования SIL для систем с низкой частотой запросов (Low demand mode)

| SIL | PFDavg (вероятность опасного отказа при запросе) | RRF (фактор снижения риска) |
|-----|--------------------------------------------------|----------------------------|
| SIL 1 | ≥ 10⁻² до < 10⁻¹ | 10–100 |
| SIL 2 | ≥ 10⁻³ до < 10⁻² | 100–1000 |
| SIL 3 | ≥ 10⁻⁴ до < 10⁻³ | 1000–10000 |
| SIL 4 | ≥ 10⁻⁵ до < 10⁻⁴ | 10000–100000 |

### Требования SIL для непрерывного режима (High demand / Continuous mode)

| SIL | PFH (вероятность опасного отказа в час) |
|-----|----------------------------------------|
| SIL 1 | ≥ 10⁻⁶ до < 10⁻⁵ |
| SIL 2 | ≥ 10⁻⁷ до < 10⁻⁶ |
| SIL 3 | ≥ 10⁻⁸ до < 10⁻⁷ |
| SIL 4 | ≥ 10⁻⁹ до < 10⁻⁸ |

### Требования к аппаратной архитектуре (IEC 61508, Hardware Fault Tolerance)

| SIL | SFF < 60% | SFF 60–90% | SFF 90–99% | SFF ≥ 99% |
|-----|-----------|------------|------------|-----------|
| SIL 1 | HFT=1 | HFT=0 | HFT=0 | HFT=0 |
| SIL 2 | — | HFT=1 | HFT=0 | HFT=0 |
| SIL 3 | — | — | HFT=1 | HFT=0 |
| SIL 4 | — | — | — | HFT=1 |

HFT = Hardware Fault Tolerance (количество отказов, которые система переносит)

## 4. Расчёт PFD и PFH

### Основные параметры
- **λDU** — интенсивность опасных необнаруженных отказов (Dangerous Undetected)
- **λDD** — интенсивность опасных обнаруженных отказов (Dangerous Detected)
- **λSD** — безопасные обнаруженные
- **λSU** — безопасные необнаруженные
- **T1** — интервал тестирования (Proof Test Interval)
- **MTTR** — среднее время восстановления
- **DC** — диагностическое покрытие (Diagnostic Coverage)
- **β** — фактор отказов по общей причине (Common Cause Factor)

### Формулы PFDavg для типовых архитектур

**1oo1 (одноканальная):**
```
PFDavg = λDU × T1 / 2
```

**1oo2 (один из двух):**
```
PFDavg = β × λDU × T1 / 2 + (1-β) × λDU² × T1² / 3
```

**2oo3 (два из трёх, мажоритарная):**
```
PFDavg = β × λDU × T1 / 2 + (1-β) × 3 × λDU² × T1² / 4
```

### Пример расчёта

**Дано:** Клапан отсечной, λDU = 5×10⁻⁷ ч⁻¹, T1 = 8760 ч (1 год), архитектура 1oo1

```
PFDavg = 5×10⁻⁷ × 8760 / 2 = 2.19×10⁻³
```

Это соответствует **SIL 2** (PFD от 10⁻³ до 10⁻²).

**Для достижения SIL 3** нужно:
- Уменьшить T1 (чаще тестировать), или
- Применить архитектуру 1oo2

## 5. Архитектуры систем безопасности

### 1oo1 (One out of One)
```
    Вход → [Канал A] → Выход
```
- Простейшая, один канал
- Любой отказ — потеря функции
- Допустима для SIL 1 (при высоком SFF)

### 1oo2 (One out of Two)
```
    Вход → [Канал A] ──┐
                        ├── OR → Выход (безопасное действие)
    Вход → [Канал B] ──┘
```
- Два параллельных канала, **любой** канал может инициировать защиту
- Высокая безопасность (оба канала должны отказать опасно)
- Повышенная частота ложных срабатываний
- **PFDavg ≈ λ² × T² / 3** (без учёта CCF)

### 2oo2 (Two out of Two)
```
    Вход → [Канал A] ──┐
                        ├── AND → Выход
    Вход → [Канал B] ──┘
```
- Оба канала должны подтвердить — тогда действие
- Низкая частота ложных срабатываний
- Но: один отказ канала = потеря защиты
- Используется для **готовности**, а не безопасности

### 2oo3 (Two out of Three) — мажоритарная
```
    Вход → [Канал A] ──┐
    Вход → [Канал B] ──┼── Мажоритарная логика (≥2 из 3) → Выход
    Вход → [Канал C] ──┘
```
- Решение по большинству (2 из 3)
- Компромисс: высокая безопасность + низкая частота ложных срабатываний
- Позволяет обслуживать один канал без останова
- Применяется для SIL 3 и SIL 4

### 1oo2D (One out of Two with Diagnostics)
```
    Вход → [Канал A + Диагностика] ──┐
                                      ├── Логика → Выход
    Вход → [Канал B + Диагностика] ──┘
```
- Каждый канал имеет встроенную самодиагностику
- При обнаружении неисправности канал переводится в безопасное состояние
- Высокий DC (≥ 90%) позволяет достичь SIL 3

## 6. Системы ПАЗ

### Определение
ПАЗ (Противоаварийная Автоматическая Защита) / СПАЗ — система, выполняющая функции противоаварийной защиты технологического объекта. В международной терминологии — SIS (Safety Instrumented System).

### Требования к ПАЗ (ФНП Ростехнадзора)

Согласно Федеральным нормам и правилам (приказ Ростехнадзора № 96 от 11.03.2013, п. 6.3):

- **6.3.1:** Системы ПАЗ обеспечивают защиту от аварийных ситуаций при отклонении параметров от допустимых значений
- **6.3.3:** ПАЗ должна быть независимой от АСУ ТП (разделение каналов)
- **6.3.5:** Методы создания определяются на стадии проектирования на основе анализа опасностей
- **6.3.6:** В проектной документации должны быть указаны перечни защит с уставками

### Принципы проектирования ПАЗ
1. **Независимость:** ПАЗ отделена от системы управления (отдельный контроллер, датчики, питание)
2. **Принцип de-energize-to-trip:** при потере питания/сигнала — переход в безопасное состояние
3. **Минимальность:** ПО ПАЗ содержит только функции безопасности (без технологического управления)
4. **Тестируемость:** возможность периодической проверки функций без останова
5. **Защита от изменений:** ограниченный доступ, аудит изменений, управление версиями

### Периодическое тестирование (Proof Test)
- **Полное функциональное тестирование** — проверка всей цепочки «датчик → логика → исполнительный механизм»
- **Частичное тестирование** — проверка отдельных компонентов (например, частичный ход клапана)
- **Интервал:** определяется расчётом SIL, типично 1–5 лет
- **Документирование:** протокол каждого теста

## 7. Сертификация ПЛК по SIL

### Требования
Контроллер для систем безопасности должен быть сертифицирован по IEC 61508 независимой организацией (TÜV, Exida, FM).

### Что проверяется
- Архитектура (HFT, SFF, DC)
- Самодиагностика (CPU, память, I/O, watchdog)
- Время реакции
- Средства разработки ПО (сертифицированные библиотеки)
- Документация (Safety Manual)

### Примеры сертифицированных контроллеров
- **HIMA HIMax** — до SIL 3 (1oo2D архитектура)
- **Honeywell Safety Manager** — SIL 3
- **Schneider Triconex Tricon** — SIL 3 (2oo3 TMR)
- **Siemens S7-400FH** — SIL 3
- **Yokogawa ProSafe-RS** — SIL 3
- **REGUL R500S** — SIL 2 (Россия, 1oo2D) [данные уточнить — reglab.ru]
- **ЭЛСИ-ТМК** — применяется в системах безопасности [уточнить SIL уровень — elesy.ru]

## 8. Российские нормы

### Основные документы

| Документ | Описание |
|----------|----------|
| ГОСТ Р МЭК 61508-1—7 | Российская адаптация IEC 61508 |
| ГОСТ Р МЭК 61511-1—3 | Российская адаптация IEC 61511 |
| ФНП «Общие правила взрывобезопасности» (Ростехнадзор, приказ № 96) | Раздел 6.3 — требования к ПАЗ |
| ГОСТ 34.602 | Техническое задание на АС |
| СТО Газпром 2-2.3-1035 | Требования к системам ПАЗ объектов газовой промышленности |

### Требования ФНП (раздел 6.3)

- ПАЗ должна обеспечивать перевод технологического процесса в безопасное состояние
- Время срабатывания ПАЗ не должно превышать значений, определённых проектом
- Системы ПАЗ должны проектироваться на базе ПЛК с аппаратным резервированием
- Логика ПАЗ не должна быть изменяемой оперативным персоналом
- Все действия ПАЗ должны архивироваться

### Экспертиза промышленной безопасности
Системы ПАЗ для опасных производственных объектов подлежат:
- Проектной экспертизе промышленной безопасности
- Вводу в эксплуатацию с участием представителей Ростехнадзора
- Периодическим проверкам (техническое диагностирование)
