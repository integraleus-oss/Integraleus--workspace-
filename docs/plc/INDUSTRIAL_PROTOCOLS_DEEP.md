## 1. Modbus RTU

### Физический уровень
- **Интерфейс:** RS-485 (2-проводный или 4-проводный), RS-232
- **Топология:** шина (multidrop) для RS-485, точка-точка для RS-232
- **Максимальная длина:** 1200 м (RS-485), 15 м (RS-232)
- **Максимум устройств:** 32 на сегменте RS-485 (до 256 с повторителями)

### Типичные настройки
- **Baud rate:** 9600, 19200, 38400, 57600, 115200 бод
- **Формат данных:** 8 бит данных, 1 стоп-бит
- **Контроль чётности:** None / Even / Odd
- **Таймаут:** 100–1000 мс (зависит от скорости и длины линии)

### Структура кадра RTU

```
┌──────────┬────────┬──────────┬───────────┬──────────┐
│ Пауза    │ Адрес  │ Функция  │  Данные   │ CRC-16   │
│ ≥3.5 сим │ 1 байт │ 1 байт   │ N байт    │ 2 байта  │
└──────────┴────────┴──────────┴───────────┴──────────┘
```

- **Адрес:** 1–247 (0 — broadcast, без ответа)
- **CRC-16:** полином x^16 + x^15 + x^2 + 1, младший байт первым
- **Пауза между кадрами:** ≥ 3.5 символа (при 9600 бод ≈ 4 мс)

### Адресация и регистры

| Тип данных | Диапазон адресов | Функции чтения | Функции записи | Доступ |
|------------|-----------------|----------------|----------------|--------|
| Coils (DO) | 00001–09999 | 01 | 05, 15 | R/W |
| Discrete Inputs (DI) | 10001–19999 | 02 | — | R |
| Input Registers (AI) | 30001–39999 | 04 | — | R |
| Holding Registers (AO) | 40001–49999 | 03 | 06, 16 | R/W |

**Внимание:** номер регистра = адрес + 1 + смещение типа. Первый Holding Register имеет номер 40001, но адрес в PDU = 0x0000.

### Основные функции

| Код | Название | Описание |
|-----|----------|----------|
| 0x01 | Read Coils | Чтение группы дискретных выходов |
| 0x02 | Read Discrete Inputs | Чтение группы дискретных входов |
| 0x03 | Read Holding Registers | Чтение группы регистров хранения |
| 0x04 | Read Input Registers | Чтение группы регистров ввода |
| 0x05 | Write Single Coil | Запись одного дискретного выхода |
| 0x06 | Write Single Register | Запись одного регистра |
| 0x0F | Write Multiple Coils | Запись нескольких дискретных выходов |
| 0x10 | Write Multiple Registers | Запись нескольких регистров |

### Пример запроса/ответа (функция 0x03)

```
Запрос:  01 03 00 00 00 02 C4 0B
         │  │  │     │     └─── CRC-16
         │  │  │     └───────── Количество регистров: 2
         │  │  └─────────────── Начальный адрес: 0
         │  └────────────────── Функция: 03 (Read Holding Registers)
         └───────────────────── Адрес устройства: 1

Ответ:   01 03 04 00 64 00 C8 DB 5A
         │  │  │  │     │     └─── CRC-16
         │  │  │  │     └───────── Регистр 2: 200 (0x00C8)
         │  │  │  └─────────────── Регистр 1: 100 (0x0064)
         │  │  └────────────────── Количество байт данных: 4
         │  └───────────────────── Функция: 03
         └──────────────────────── Адрес: 1
```

### Диагностика неисправностей
- **Нет ответа:** проверить адрес, скорость, чётность, терминатор линии
- **Ошибка CRC:** помехи в линии, плохой контакт, длина кабеля
- **Exception codes:** 01=недопустимая функция, 02=недопустимый адрес, 03=недопустимое значение

## 2. Modbus TCP

### Физический уровень
- **Интерфейс:** Ethernet 10/100/1000 Мбит
- **Порт:** TCP/502
- **Топология:** звезда (через коммутатор)

### Отличия от RTU
- Нет CRC (надёжность обеспечивает TCP)
- Нет пауз между кадрами
- Вместо адреса устройства — Unit ID (для шлюзов Modbus TCP→RTU)
- Добавлен заголовок MBAP (Modbus Application Protocol Header)

### Структура MBAP Header + PDU

```
┌──────────────────────────────────────────┬──────────────────┐
│           MBAP Header (7 байт)           │    PDU           │
├──────────┬──────────┬─────────┬──────────┼────────┬─────────┤
│Trans. ID │Protocol  │ Length  │ Unit ID  │Function│ Data    │
│ 2 байта  │ 2 байта  │ 2 байта│ 1 байт   │ 1 байт │ N байт  │
│          │ (0x0000) │        │          │        │         │
└──────────┴──────────┴─────────┴──────────┴────────┴─────────┘
```

- **Transaction ID:** клиент присваивает для сопоставления запрос/ответ
- **Protocol ID:** всегда 0x0000 для Modbus
- **Length:** количество байт после Length (Unit ID + PDU)

## 3. OPC UA

### Архитектура
- **Тип:** клиент-серверный протокол, возможен pub/sub (OPC UA Part 14)
- **Транспорт:** TCP (порт 4840), HTTPS, WebSocket
- **Безопасность:** шифрование (AES-128/256), аутентификация по сертификатам X.509, подпись сообщений

### Адресное пространство
Информационная модель OPC UA — это граф узлов (Nodes), связанных ссылками (References).

**Типы узлов:**
- **Object** — объект реального мира (насос, клапан)
- **Variable** — значение (температура, давление)
- **Method** — вызываемая функция (запуск, останов)
- **ObjectType, VariableType** — типы
- **ReferenceType** — типы ссылок
- **DataType** — типы данных
- **View** — фильтры адресного пространства

**NodeId** — уникальный идентификатор: `ns=<namespace>;s=<string>` или `ns=<namespace>;i=<integer>`

### Сервисы OPC UA
- **Discovery:** FindServers, GetEndpoints
- **Session:** CreateSession, ActivateSession
- **Read/Write:** Read, Write
- **Subscription:** CreateSubscription, Publish (данные по изменению)
- **Method:** Call
- **Browse:** Browse (навигация по адресному пространству)
- **HistoryRead:** чтение архивных данных

### Безопасность
- **Режимы:** None, Sign, SignAndEncrypt
- **Политики:** Basic256Sha256, Aes128_Sha256_RsaOaep, Aes256_Sha256_RsaPss
- **Аутентификация:** Anonymous, UserName/Password, X.509 Certificate

## 4. Profibus DP

### Физический уровень
- **Интерфейс:** RS-485 (Profibus DP), MBP (Manchester Coded, Bus Powered — Profibus PA)
- **Скорость:** 9.6 кбит/с — 12 Мбит/с (зависит от длины)
- **Длина сегмента:** 100 м (12 Мбит) — 1200 м (9.6 кбит)
- **Макс. устройств:** 126 (адреса 0–125, 126 — зарезервирован)
- **Разъёмы:** DB-9 (стандарт)

### Архитектура Master/Slave
- **DP Master Class 1:** контроллер, циклический обмен
- **DP Master Class 2:** инженерная станция, диагностика
- **DP Slave:** модули ввода-вывода, приводы, датчики

### GSD-файлы
GSD (General Station Description) — текстовый файл с описанием slave-устройства:
- Идентификатор устройства (Ident_Number)
- Поддерживаемые скорости
- Конфигурация модулей ввода/вывода
- Диагностические данные

### Типичная конфигурация
1. Импортировать GSD-файл в среду (TIA Portal, CoDeSys)
2. Добавить slave на шину, задать адрес
3. Сконфигурировать модули ввода/вывода
4. Назначить адреса I/O в адресном пространстве контроллера
5. Скомпилировать и загрузить

### Диагностика
- Индикаторы BF (Bus Fault) на устройствах
- Диагностические телеграммы (Slot 0, диагностический PDU)
- Типичные проблемы: терминатор, адрес, скорость, качество кабеля

## 5. PROFINET

### Физический уровень
- **Интерфейс:** Ethernet 100 Мбит (Fast Ethernet)
- **Кабель:** Cat5e, оптика (для больших расстояний)
- **Топология:** звезда, линейная (daisy-chain), кольцо (MRP)

### Классы
- **PROFINET RT (Real-Time):** время цикла ≥ 1 мс, используется стандартный Ethernet
- **PROFINET IRT (Isochronous Real-Time):** время цикла < 1 мс, требуется специальный ASIC

### GSDML-файлы
Аналог GSD для Profibus, но в формате XML. Содержит описание IO-Device: модули, подмодули, параметры, диагностика.

### Архитектура
- **IO Controller** — ПЛК, управляет IO-Device
- **IO Device** — полевые устройства
- **IO Supervisor** — инженерная станция (диагностика, параметрирование)

## 6. МЭК 60870-5-101

### Назначение
Протокол телемеханики для последовательных каналов связи. Применяется в энергетике, нефтегазе, водоснабжении.

### Физический уровень
- **Интерфейс:** RS-232, RS-485, оптика, радиоканал
- **Скорость:** 300–115200 бод
- **Режимы:** симметричный (balanced) и несимметричный (unbalanced)

### Структура кадра
Формат FT1.2 (стандарт МЭК 870-5-1):

```
Фиксированная длина:        Переменная длина:
┌──────┐                     ┌──────┐
│ 10H  │ ← Start            │ 68H  │ ← Start
│ C    │ ← Control field     │ L    │ ← Length
│ A    │ ← Address           │ L    │ ← Length (повтор)
│ CS   │ ← Checksum          │ 68H  │ ← Start (повтор)
│ 16H  │ ← Stop              │ C    │ ← Control field
└──────┘                     │ A    │ ← Address
                             │ ASDU │ ← Application data
                             │ CS   │ ← Checksum
                             │ 16H  │ ← Stop
                             └──────┘
```

### ASDU (Application Service Data Unit)
- **Type ID** — тип информации (1 байт)
- **Qualifier** — количество объектов, SQ-бит
- **Cause of Transmission** — причина передачи (циклический, спорадический, команда и т.д.)
- **ASDU Address** — адрес станции
- **Information Objects** — данные

### Основные типы ASDU

| TypeID | Описание | Направление |
|--------|----------|-------------|
| 1 | Одноэлементная информация (Single Point) | Monitor |
| 3 | Двухэлементная информация (Double Point) | Monitor |
| 9 | Нормализованное измеренное значение | Monitor |
| 13 | Измеренное значение, короткое с плавающей точкой | Monitor |
| 30 | Одноэл. с меткой времени CP56Time2a | Monitor |
| 45 | Одноэлементная команда | Control |
| 46 | Двухэлементная команда | Control |
| 50 | Команда уставки, нормализованная | Control |
| 100 | Команда опроса (General Interrogation) | Control |
| 103 | Команда синхронизации часов | Control |

## 7. МЭК 60870-5-104

### Отличия от МЭК 101
- Транспорт: **TCP/IP** вместо последовательного канала
- Порт: **TCP/2404**
- Не нужен адрес канального уровня (заменён TCP-соединением)
- Добавлен уровень APCI (Application Protocol Control Information)

### Структура APDU

```
APDU = APCI + ASDU

APCI (6 байт):
┌──────┬────────┬──────────────┐
│ 68H  │ Length │ Control      │
│ Start│ 1 байт│ 4 байта      │
└──────┴────────┴──────────────┘
```

### Типы кадров APCI

| Формат | Описание | Назначение |
|--------|----------|------------|
| I-формат | Information transfer | Передача ASDU (данные) |
| S-формат | Supervisory | Квитирование (подтверждение приёма) |
| U-формат | Unnumbered | Управление соединением (STARTDT, STOPDT, TESTFR) |

### Параметры соединения
- **k** — максимум неквитированных I-кадров (по умолч. 12)
- **w** — порог квитирования (по умолч. 8)
- **t0** — таймаут установления соединения (30 с)
- **t1** — таймаут ожидания ответа (15 с)
- **t2** — таймаут квитирования (10 с)
- **t3** — таймаут проверки связи TESTFR (20 с)

## 8. МЭК 61850

### Назначение
Стандарт для автоматизации электрических подстанций. Объектно-ориентированная модель данных.

### Протоколы в составе МЭК 61850
- **MMS (Manufacturing Message Specification)** — клиент-серверный обмен поверх TCP/IP
- **GOOSE (Generic Object Oriented Substation Event)** — быстрая передача событий по Ethernet (multicast), время < 4 мс
- **Sampled Values (SV)** — передача мгновенных значений тока/напряжения
- **SCL (Substation Configuration Language)** — XML-файлы конфигурации

### SCL-файлы

| Тип файла | Расширение | Содержание |
|-----------|------------|------------|
| ICD | .icd | Описание возможностей IED (от производителя) |
| SSD | .ssd | Описание подстанции (однолинейная схема) |
| SCD | .scd | Полная конфигурация подстанции |
| CID | .cid | Конфигурация конкретного IED (для загрузки) |

### Модель данных
- **IED** (Intelligent Electronic Device) → **Logical Device** → **Logical Node** → **Data Object** → **Data Attribute**
- Пример: `IED1/LLN0.GGIO1.Ind1.stVal` — значение дискретного сигнала

## 9. HART

### Физический уровень
- **Принцип:** цифровой сигнал FSK (1200/2200 Гц) наложен на аналоговый токовый сигнал 4–20 мА
- **Скорость:** 1200 бод
- **Режимы:** точка-точка (polling address 0) и multidrop (address 1–63, ток фиксирован 4 мА)
- **Кабель:** обычная 2-проводная линия 4–20 мА

### Структура кадра HART

```
┌────────┬───────┬────────┬────────────┬────────┬────────┬──────┐
│Preamble│ Start │ Address│ Command    │ Byte   │ Data   │ Check│
│ 5-20   │ 1 байт│ 1 или  │ 1 байт     │ Count  │ 0-25   │ Sum  │
│ 0xFF   │       │ 5 байт │            │ 1 байт │ байт   │1 байт│
└────────┴───────┴────────┴────────────┴────────┴────────┴──────┘
```

### Группы команд

| Группа | Номера | Описание |
|--------|--------|----------|
| Universal | 0–30 | Обязательные для всех устройств (ID, PV, диагностика) |
| Common Practice | 32–126 | Типовые (настройка диапазона, единиц измерения) |
| Device Specific | 128–253 | Специфичные для конкретного устройства |

### Основные универсальные команды
- **Cmd 0:** Read Unique Identifier
- **Cmd 1:** Read Primary Variable (PV)
- **Cmd 2:** Read Current and Percent of Range
- **Cmd 3:** Read Dynamic Variables and Current
- **Cmd 6:** Write Polling Address
- **Cmd 15:** Read Device Information

### Настройка
- HART-коммуникатор (портативный терминал) или HART-модем + ПО
- Подключение параллельно токовой петле (резистор ≥ 250 Ом)
- Настраиваемые параметры: диапазон измерения, демпфирование, единицы, тип линеаризации

## 10. CAN / CANopen

### CAN (Controller Area Network)
- **Физический уровень:** дифференциальная пара (CAN_H, CAN_L), витая пара
- **Скорость:** до 1 Мбит/с (до 40 м), 125 кбит/с (до 500 м)
- **Кадр:** 11 бит ID (стандартный) или 29 бит (расширенный), до 8 байт данных
- **Арбитраж:** побитовый, приоритет по ID (меньший ID — выше приоритет)
- **Обнаружение ошибок:** CRC-15, Stuff-bit, проверка формы

### CANopen (протокол верхнего уровня)
- **Профиль:** CiA 301 (коммуникационный), CiA 401 (I/O), CiA 402 (приводы)
- **Объектный словарь (OD):** стандартизированная структура параметров устройства
- **PDO (Process Data Object):** циклический обмен данными, без подтверждения
- **SDO (Service Data Object):** обмен параметрами конфигурации, с подтверждением
- **NMT (Network Management):** управление состояниями узлов

### Настройка CANopen
1. Установить baud rate на всех узлах (125, 250, 500, 1000 кбит/с)
2. Назначить Node-ID каждому устройству (1–127)
3. Сконфигурировать PDO mapping (какие данные передавать)
4. Настроить EDS-файл (Electronic Data Sheet) — аналог GSD

## 11. EtherCAT

### Принцип работы
Кадр Ethernet проходит через все slave-устройства **последовательно** (on-the-fly processing). Каждый slave считывает адресованные ему данные и вставляет свои «на лету», без буферизации. Кадр возвращается к master-у.

### Физический уровень
- **Интерфейс:** Ethernet 100 Мбит (IEEE 802.3)
- **Кабель:** Cat5e (до 100 м между узлами)
- **Топология:** линейная (daisy-chain), звезда, дерево, кольцо (для резервирования)

### Внутренняя логическая топология
Всегда **кольцевая** — даже при физической линейной топологии. Последний slave «заворачивает» кадр обратно через второй порт. При физическом кольце — автоматическое резервирование при обрыве.

### Характеристики
- **Время цикла:** до 12.5 мкс (для 100 устройств)
- **Джиттер:** < 1 мкс
- **Макс. устройств:** 65535 на сегменте
- **Синхронизация:** Distributed Clocks (DC), точность < 100 нс

### Типы кадров EtherCAT
- **EtherCAT Frame:** стандартный Ethernet frame с EtherType 0x88A4
- **Datagram:** единица данных внутри EtherCAT frame (может быть несколько)
- **Адресация:** позиционная, настроенная, логическая (FMMU mapping)

### Применение
- Сервоприводы и приводы (CiA 402 over EtherCAT — CoE)
- Модули ввода-вывода
- Высокоскоростные системы управления движением
- Робототехника
