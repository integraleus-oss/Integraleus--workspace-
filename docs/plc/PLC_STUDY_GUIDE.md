## 1. Архитектура ПЛК

### 1.1 Что такое ПЛК

**Программируемый логический контроллер (ПЛК)** (англ. Programmable Logic Controller, PLC) — цифровая электронная система, предназначенная для применения в производственной среде, которая использует программируемую память для внутреннего хранения инструкций по реализации таких специальных функций, как логика, установление последовательности, согласование по времени, счёт и арифметические действия для контроля посредством цифрового или аналогового ввода/вывода данных различных видов машин или процессов.

ПЛК — устройства, предназначенные для работы в системах реального времени. В качестве основного режима работы выступает длительное автономное использование, зачастую в неблагоприятных условиях окружающей среды, без серьёзного обслуживания и практически без вмешательства человека.

> *Источник: https://ru.wikipedia.org/wiki/Программируемый_логический_контроллер*

### 1.2 История создания

- **1968 г.** — компания General Motors (GM) инициировала проект Standard Machine Controller с целью замены традиционных релейных систем управления станками. Запрос направлен компаниям Allen-Bradley, Digital Equipment Corporation (DEC), Century Detroit и Bedford Associates.
- **1968–1969 гг.** — команда Bedford Associates под руководством Ричарда Морли разработала **Modicon 084** — первый в мире программируемый контроллер. Modicon = MOdular DIgital CONtroller.
- **1971 г.** — инженеры Allen-Bradley под руководством Одо Жозефа Стругера разработали Bulletin 1774 PLC, впервые использовав термин **ПЛК** вместо ПК.
- **1973 г.** — Modicon представила усовершенствованную модель Modicon 184.
- **~1980 г.** — в СССР разработан контроллер «Электроника К1-20» на базе БИС серий К580.
- Логика первых ПЛК программировалась на языке релейных диаграмм (LD), что было привычно для электриков и инженеров, обслуживавших релейные системы.

> *Источник: https://ru.wikipedia.org/wiki/Программируемый_логический_контроллер*

### 1.3 Компоненты ПЛК

| Компонент | Описание |
|-----------|----------|
| **Центральный процессор (CPU)** | «Мозг» системы, исполняет программу управления |
| **Модули ввода (DI/AI)** | Принимают сигналы от датчиков, кнопок, выключателей (24 В цифровой, 0–10 В / 4–20 мА аналоговый) |
| **Модули вывода (DO/AO)** | Управляют исполнительными устройствами: клапаны, реле, частотные преобразователи, двигатели |
| **Источник питания** | Обычно 24 В постоянного тока, обеспечивает энергией все модули |
| **Коммуникационные модули** | Интерфейсы RS-485, RS-232, Ethernet, CAN для связи с другими устройствами |
| **Память** | Хранит программу и данные (ОЗУ, ПЗУ, флэш) |

### 1.4 Цикл сканирования (Scan Cycle)

Любой ПЛК работает в **циклическом режиме**. Цикл состоит из трёх основных фаз:

1. **Чтение входов** — сбор данных с модулей ввода (считывание сигналов от датчиков)
2. **Выполнение программы** — обработка данных по заданному алгоритму управления в процессоре
3. **Запись выходов** — передача результатов на модули вывода (команды на исполнительные механизмы)

Этот цикл повторяется с высокой скоростью — обычно каждые **1–20 мс**. Величина контроллерного цикла зависит от времени исполнения программы и быстродействия процессорного модуля.

Разные фрагменты программы могут исполняться с разной частотой. Например, один фрагмент может сканировать концевой датчик 100 раз в секунду, а второй — датчик температуры раз в 10 секунд.

### 1.5 Типы ПЛК

| Тип | Описание |
|-----|----------|
| **Моноблочные** | Один корпус содержит CPU и базовый набор I/O, может расширяться по шине. Компромисс между гибкостью и компактностью |
| **Модульные** | Масштабируемые системы с отдельными модулями CPU, аналогового и цифрового I/O, интерфейсами. Каждый модуль подключается к системной шине. Для гибких, высоконагруженных систем |
| **Программируемые реле** | Упрощённая форма ПЛК с базовой логикой (таймеры, триггеры, логические операции). Для простых повторяющихся задач |
| **Распределённые** | Модули I/O размещены на удалённых узлах, связаны с центральным CPU по промышленной сети |
| **Панельные** | ПЛК со встроенной панелью оператора (HMI) в едином корпусе |

> *Источник: https://www.invt.su/articles/programmiruemyy-logicheskiy-kontroller-chto-eto-i-zachem-nuzhen/*

### 1.6 Резервирование

| Схема | Описание |
|-------|----------|
| **1oo1** | Один канал, одного достаточно для срабатывания. Максимально SIL 2 |
| **1oo2** | Два канала, одного достаточно для срабатывания. Обеспечивает высокую надёжность — отказ одного канала не приводит к отказу системы. Максимально SIL 3 |
| **2oo3 (TMR)** | Три канала (тройное модульное резервирование), двух достаточно для срабатывания. Обеспечивает надёжность и защиту от ложных срабатываний. Максимально SIL 4 |
| **1oo2D** | Два канала с диагностикой. Максимально SIL 3 |
| **Горячая замена (Hot Swap)** | Возможность замены модулей без остановки работы системы |

## 2. Стандарт IEC 61131

### 2.1 Общие сведения

**IEC 61131** (МЭК 61131) — международный стандарт, определяющий требования к программируемым логическим контроллерам. Первоначально назывался IEC 1131, в 1997 г. МЭК перешёл на новую систему обозначений и добавилась цифра «6».

Продвижением стандарта занимается организация **PLCopen** (http://www.plcopen.org).

### 2.2 Части стандарта

| Часть | Название |
|-------|----------|
| IEC 61131-1 | Общие сведения |
| IEC 61131-2 | Требования к оборудованию и тестированию |
| **IEC 61131-3** | **Языки программирования** |
| IEC 61131-4 | Руководство пользователя |
| IEC 61131-5 | Коммуникации |
| IEC 61131-6 | Функциональная безопасность |
| IEC 61131-7 | Программирование нечёткой логики |
| IEC 61131-8 | Руководство по применению языков |
| IEC 61131-9 | Цифровой коммуникационный интерфейс одного входа/выхода (IO-Link) |
| IEC 61131-10 | Формат обмена XML для PLC |

> *Источник: https://ru.wikipedia.org/wiki/IEC_61131-3*

### 2.3 IEC 61131-3: Языки программирования

Стандарт IEC 61131-3 устанавливает **5 языков программирования** ПЛК — три графических и два текстовых:

| Язык | Полное название | Тип |
|------|----------------|-----|
| **LD** | Ladder Diagram (релейно-контактные схемы) | Графический |
| **FBD** | Function Block Diagram (диаграммы функциональных блоков) | Графический |
| **SFC** | Sequential Function Chart (последовательные функциональные схемы) | Графический |
| **ST** | Structured Text (структурированный текст) | Текстовый |
| **IL** | Instruction List (список инструкций) | Текстовый |

**Редакции стандарта:**
- 1-я редакция: 1993 г. (IEC 61131-3:1993)
- 2-я редакция: 2003 г. (IEC 61131-3:2003)
- 3-я редакция: 2013 г. (IEC 61131-3:2013 / ГОСТ Р МЭК 61131-3-2016)

**Нововведения 3-й редакции:**
- Классы, методы, интерфейсы (ООП)
- Объектно-ориентированные функциональные блоки
- Пространства имён
- Ссылки и операции со ссылками
- Массивы с изменяемой длиной
- Операторы Continue в ST
- Контакты для сравнения в LD

**Устаревшее (deprecated):**
- Язык IL (Instruction List) — исключён в 3-й редакции
- Восьмеричные литералы

> *Источники: https://ru.wikipedia.org/wiki/IEC_61131-3, https://www.reallab.ru/bookasutp/9-programmnoe-obespechenie/9-3-sistemi-programmirovaniya-mek-61131-3/*

### 2.4 Ключевые принципы стандарта

- Вся программа разбивается на **Program Organization Units (POU)** — функции, функциональные блоки и программы
- **Строгая типизация данных** — ошибки обнаруживаются до исполнения
- Поддержка **параллельного выполнения** фрагментов программы с разной частотой
- Поддержка **структур данных** для описания разнородных данных
- **Совместное использование** всех 5 языков в одном проекте
- **Переносимость** программ между совместимыми контроллерами

> *Источник: https://www.reallab.ru/bookasutp/9-programmnoe-obespechenie/9-3-sistemi-programmirovaniya-mek-61131-3/*

## 3. Языки программирования IEC 61131-3

### 3.1 LD (Ladder Diagram) — Релейные диаграммы

**Описание и назначение:**
LD — графический язык, реализующий структуры электрических цепей. Представляет собой программную реализацию электрических схем на базе электромагнитных реле. Появился в начале 1970-х годов в США, когда релейная автоматика начала вытесняться контроллерами. Интуитивно понятен людям, знакомым с электротехникой.

**Синтаксис:**
Графически LD-диаграмма представлена в виде двух вертикальных шин питания, между которыми расположены цепи, образованные соединением контактов. Нагрузкой каждой цепи служит «катушка» реле. Основные элементы:
- **Контакт нормально разомкнутый** `--| |--` — замыкается при TRUE
- **Контакт нормально замкнутый** `--|/|--` — размыкается при TRUE
- **Катушка** `--( )--` — устанавливает выходную переменную
- Контакты могут включаться **последовательно** (логическое И) и **параллельно** (логическое ИЛИ)

LD позволяет включать в цепи функциональные блоки, написанные на других языках.

**Где применяется:**
- Простая дискретная логика (блокировки, пуск/стоп)
- Замена релейных схем автоматики
- Проверка работоспособности схем на реле

**Плюсы:**
- Интуитивно понятен электрикам и обслуживающему персоналу
- Наиболее распространённый язык в мире
- Простота отладки — видно путь прохождения сигнала
- В некоторых средах разработки — единственный доступный язык

**Минусы:**
- Не поддерживает подпрограммы, инкапсуляцию, структурирование
- Сложные вычисления невозможны
- Маленькая часть программы умещается на экране
- Не подходит для сложных алгоритмов

**Производители:** Поддерживается всеми основными производителями ПЛК (Siemens, Allen-Bradley/Rockwell, Schneider Electric, ОВЕН, ABB, Mitsubishi, Omron и др.)

### 3.2 FBD (Function Block Diagram) — Функциональные блоки

**Описание и назначение:**
FBD — графический язык, наиболее удобный для программирования процессов прохождения сигналов через функциональные блоки. Диаграмма FBD напоминает принципиальную (или функциональную) схему электронного устройства на микросхемах. Удобен для схемотехников и специалистов по САУ.

**Синтаксис:**
Программа составляется из **функциональных блоков** — прямоугольников со входами (слева) и выходами (справа). Блоки соединяются «проводниками», которые могут передавать значения любых типов (логические, аналоговые, время и т.п.). Основные элементы:
- Стандартные блоки: AND, OR, NOT, таймеры (TON, TOF), счётчики (CTU, CTD), ПИД-регулятор, триггеры (RS, SR), генераторы импульсов
- Пользовательские функциональные блоки

**Пример** (текстовое описание):
```
[Input1]--+
           |--[AND]--[Output]
[Input2]--+
```

**Где применяется:**
- Описание «жёсткой логики»
- Замкнутые контуры систем управления (ПИД-регулирование)
- Обработка аналоговых сигналов
- Создание библиотек типовых блоков

**Плюсы:**
- Наглядное отображение прохождения и преобразования сигналов
- Инкапсуляция данных и методов в блоках
- Многократное использование блоков
- Удобен для создания библиотек

**Минусы:**
- Не поддерживает наследование и полиморфизм
- Громоздкие диаграммы при сложных алгоритмах
- Реализация в разных средах может отличаться (в CoDeSys FBD состоит из цепей, как LD)

**Производители:** Siemens (STEP 7 / TIA Portal), CODESYS, Schneider Electric (Unity Pro), ОВЕН, ABB, Yokogawa, Mitsubishi, Omron

### 3.3 ST (Structured Text) — Структурированный текст

**Описание и назначение:**
ST — текстовый язык высокого уровня, очень похожий на Паскаль. Разработан специально для программирования ПЛК. Предназначен для сложных математических вычислений, описания сложных функций, функциональных блоков и программ.

**Синтаксис:**

```pascal
(* Пример: управление током и мотором *)
IF Voltage > 220 THEN
    Current := Current - 10;   (* Уменьшить ток на 10 *)
ELSE
    Current := 50;
    Speed := ON;               (* Включить мотор *)
END_IF;

(* Объявление переменных *)
VAR
    iVar : INT;
    bVar : BOOL;
    tVar : TIME;
END_VAR

(* Цикл FOR *)
FOR i := 1 TO 10 BY 2 DO
    k := k * 2;
END_FOR;

(* Цикл WHILE *)
WHILE k > 0 DO
    i := i + k;
    k := k - 1;
END_WHILE;

(* Конструкция CASE *)
CASE k OF
    1:      k := k * 10;
    2..5:   k := k * 5;
    6, 9..20: k := k - 1;
ELSE
    k := 0;
END_CASE;
```

**Основные конструкции:** присвоение (`:=`), IF-ELSE, FOR, WHILE, CASE, вызовы функций и ФБ.

**Особенности:**
- Ключевые слова — в ВЕРХНЕМ регистре (IF, THEN, END_IF, VAR и т.д.)
- Идентификаторы — не чувствительны к регистру
- Точка с запятой в конце выражения
- Нет автоматического приведения типов

**Где применяется:**
- Сложные математические вычисления
- Алгоритмы регулирования
- Обработка данных, протоколы связи
- Везде, где нужен разветвлённый алгоритм

**Плюсы:**
- Наиболее гибкий и мощный из языков МЭК
- Понятен программистам (особенно знакомым с Паскалем)
- Поддерживает все конструкции структурного программирования
- Компактный код

**Минусы:**
- Менее наглядный для электриков и технологов
- Сложнее для визуальной отладки «на лету»

**Производители:** Поддерживается всеми современными средами: CODESYS, Siemens (STEP 7/TIA), Schneider Electric, ОВЕН, B&R, Beckhoff (TwinCAT), ABB и др.

### 3.4 IL (Instruction List) — Список инструкций

**Описание и назначение:**
IL — текстовый язык, напоминающий ассемблер. Используется для реализации функций, функциональных блоков и программ. **Устарел и исключён в 3-й редакции стандарта IEC 61131-3 (2013 г.).**

**Синтаксис:**
В основе — понятие **аккумулятора** и **переходов по меткам**. Всего 24 оператора.

```
     LD    Voltage      (* Загрузить Voltage в аккумулятор *)
     GT    220          (* Если > 220 *)
     JMPCN M1           (* Перейти к метке, если не верно *)
     LD    Current      (* Загрузить Current *)
     SUB   10           (* Вычесть 10 *)
     ST    Current      (* Сохранить результат *)
M1:  LD    0            (* Загрузить 0 *)
     ST    Out          (* Присвоить Out *)
```

**Основные операторы:** LD, LDN, ST, STN, S, R, AND, OR, XOR, ADD, SUB, MUL, DIV, GT, GE, EQ, NE, LE, LT, JMP, JMPC, JMPCN

**Где применяется:**
- Оптимизация критических секций кода
- Небольшие задачи с малым количеством ветвлений
- Устаревшие проекты

**Плюсы:**
- Простота изучения
- Минимальный код — максимальная оптимизация
- Полный контроль над выполнением

**Минусы:**
- **Устарел** (deprecated в IEC 61131-3:2013)
- Не подходит для сложных алгоритмов
- Плохая читаемость
- Нет структурирования

**Производители:** Поддерживается в CODESYS, Beremiz, некоторых старых средах. Новые среды постепенно отказываются от IL.

> *Источники: https://www.reallab.ru/bookasutp/9-programmnoe-obespechenie/9-3-sistemi-programmirovaniya-mek-61131-3/, https://bric-beremiz.readthedocs.io/ru/latest/mek61131_3.html, https://ru.wikipedia.org/wiki/IEC_61131-3*

### 3.5 SFC (Sequential Function Chart) — Последовательные функциональные схемы

**Описание и назначение:**
SFC — графический высокоуровневый язык, созданный на базе математического аппарата **сетей Петри**. Предназначен для программирования последовательности выполнения действий, когда действия должны выполняться в заданные моменты времени или при наступлении событий.

По сути — это вспомогательное средство для **структурирования программ** на верхнем уровне абстракции.

**Синтаксис:**
Программа состоит из **шагов** (прямоугольники) и **условий переходов** (жирные перечёркивающие линии):
- **Начальный шаг** — двойной прямоугольник
- **Шаги** — прямоугольники с именем действия
- **Переходы** — условия перехода между шагами
- Программа выполняется **сверху вниз**
- Каждый шаг может содержать программу на любом из МЭК-языков

**Пример (текстовое описание):**
```
[= INIT =]          ← начальный шаг
    |
  (Start)            ← условие перехода
    |
[Наполнение]         ← шаг
    |
  (Level > 80%)      ← условие перехода
    |
[Этап_1]             ← шаг
    |
  (Timer > 60s)
    |
[Выгрузка]
```

**Где применяется:**
- Описание технологических последовательностей (рецепты, пакетные процессы)
- Управление многошаговыми процессами: «Старт → Наполнение → Обработка → Выгрузка»
- Алгоритмы с состояниями (автоматическое соединение модема, протокол связи)

**Плюсы:**
- Высокий уровень абстракции
- Наглядное описание последовательных процессов
- Комбинируется с другими языками
- Основан на математическом аппарате (сети Петри)

**Минусы:**
- Не самостоятельный «язык» в полном смысле — вспомогательное средство
- Для реализации каждого шага нужен другой язык
- Сложные параллельные ветвления требуют аккуратного проектирования

**Производители:** CODESYS, Siemens (GRAPH в STEP 7), Schneider Electric, B&R, Beremiz, ISaGRAF

> *Источники: https://www.reallab.ru/bookasutp/9-programmnoe-obespechenie/9-3-sistemi-programmirovaniya-mek-61131-3/, https://finestart.school/media/programming_languages, https://ru.wikipedia.org/wiki/IEC_61131-3*

### 3.6 CFC (Continuous Function Chart) — расширение FBD

**Описание и назначение:**
CFC (Continuous Function Chart) — расширение языка FBD, не входящее в стандарт IEC 61131-3, но широко поддерживаемое средами разработки. В отличие от стандартного FBD, где блоки располагаются в цепях (как в LD), в CFC блоки размещаются **свободно** на рабочей поверхности и соединяются линиями в произвольном порядке.

**Отличия от FBD:**
- Блоки могут находиться в любой части экрана
- Нет жёсткой привязки к цепям
- Более наглядное представление сложных схем
- Свободное соединение входов и выходов

**Где применяется:**
- Сложные системы непрерывного управления
- Схемы с большим количеством обратных связей
- Визуализация алгоритмов для инженеров САУ

**Производители:** CODESYS (CFC-редактор), Siemens (CFC в PCS 7), Schneider Electric (Unity Pro), ABB (800xA), Yokogawa (CENTUM VP)

> *Источник: https://asu-app.ru/development/plc/yazyk-programmirovaniya-fbd.php*

## 5. Промышленные протоколы связи

### 5.1 Modbus RTU/TCP

- **Назначение:** Один из самых распространённых промышленных протоколов обмена данными
- **Modbus RTU:** Работает по RS-485/RS-232, двоичный формат, контрольная сумма CRC
- **Modbus TCP:** Работает по Ethernet (TCP/IP), инкапсуляция Modbus-кадров в TCP-пакеты
- **Архитектура:** Master-Slave (ведущий-ведомый)
- **Применение:** Опрос датчиков, управление частотными преобразователями, связь SCADA с ПЛК
- **Особенности:** Простой, открытый, широко поддерживаемый. Медленный при циклическом опросе — можно потерять быстрые события.

> *Источники: https://rzia.ru/topic6870-protokoly-peredachi-dannykh-kakoi-udobnei.html, https://www.souz-pribor.ru/articles/kak-rabotaet-plk-kontroller/, https://www.invt.su/articles/programmiruemyy-logicheskiy-kontroller-chto-eto-i-zachem-nuzhen/*

### 5.2 OPC UA

- **Назначение:** Универсальная платформа для обмена данными между промышленными системами
- **Полное название:** Open Platform Communications Unified Architecture
- **Особенности:** Кроссплатформенный, поддерживает шифрование и аутентификацию, моделирование данных, историю, события
- **Применение:** Интеграция различных систем автоматизации, связь MES/ERP с АСУ ТП, IoT/Industry 4.0
- **Предшественник:** OPC DA (COM/DCOM, только Windows)

> *Источники: https://www.cta.ru/articles/cta/obzory/promyshlennye-seti/124402/, https://smis-expert.com/blog/protokol-modbus-dlya-raboty-opc-serverov/*

### 5.3 Profibus DP

- **Назначение:** Полевая шина для связи ПЛК с распределённой периферией
- **Стандарт:** IEC 61158
- **Скорость:** до 12 Мбит/с
- **Физическая среда:** RS-485 (экранированная витая пара)
- **Применение:** Связь CPU с модулями удалённого I/O, частотными преобразователями, приводами
- **Производитель:** Разработан Siemens и ряд компаний

> *Источник: https://www.cta.ru/articles/cta/obzory/promyshlennye-seti/124402/*

### 5.4 Profinet

- **Назначение:** Промышленный Ethernet-протокол реального времени
- **Стандарт:** IEC 61158, IEC 61784
- **Особенности:** Real-time (RT) и изохронный реальном времени (IRT), работает поверх стандартного Ethernet
- **Применение:** Замена Profibus DP в современных системах, управление приводами, распределённый I/O
- **Производитель:** Siemens / PI (Profibus & Profinet International)

> *Требует дополнительных источников для детального описания.*

### 5.5 МЭК 60870-5-101/104

- **Назначение:** Протоколы телемеханики для передачи данных в энергетике и распределённых системах
- **МЭК 101:** Работает по последовательным каналам (RS-232/485)
- **МЭК 104:** Работает по TCP/IP (Ethernet)
- **Особенности:** Спонтанная передача данных (при изменении), метки времени, качество данных. Протоколы для телемеханики, оптимизированы для передачи событий без потерь.

### 5.6 МЭК 61850

- **Назначение:** Стандарт для коммуникаций на цифровых подстанциях
- **Особенности:** Объектно-ориентированная модель данных, GOOSE-сообщения (горизонтальная связь между устройствами в реальном времени), MMS (Manufacturing Message Specification)
- **Применение:** Цифровые подстанции, РЗА (релейная защита и автоматика)

- **Назначение:** Протокол для связи с интеллектуальными полевыми приборами

### 5.8 CAN/CANopen

- **Назначение:** Последовательная шина для связи узлов в режиме реального времени

- **CANopen:** Высокоуровневый протокол на базе CAN для промышленной автоматизации
- **Применение:** Мобильная техника, транспорт, малые системы автоматизации

> *Источник: https://www.invt.su/articles/programmiruemyy-logicheskiy-kontroller-chto-eto-i-zachem-nuzhen/*

### 5.9 EtherCAT

- **Назначение:** Высокоскоростной Ethernet-протокол реального времени
- **Особенности:** Метод «on the fly» — кадр Ethernet обрабатывается каждым узлом по мере прохождения, не задерживаясь. Время цикла — микросекунды.
- **Применение:** Высокоскоростное управление приводами, робототехника, прецизионные станки
- **Производитель:** Beckhoff / ETG (EtherCAT Technology Group)

## 6. Функциональная безопасность

### 6.1 IEC 61508 / ГОСТ Р МЭК 61508

**IEC 61508** — основополагающий международный стандарт в области функциональной безопасности электрических, электронных и программируемых электронных систем, связанных с безопасностью.

Российский эквивалент — **ГОСТ Р МЭК 61508** (серия стандартов, ~600 страниц текста).

**Структура стандарта:**

| Часть | Название |
|-------|----------|
| Часть 1 | Общие требования (основные концепции и терминология) |
| Часть 2 | Требования к системам (аппаратные средства) |
| Часть 3 | Требования к программному обеспечению |
| Часть 4 | Термины и определения |
| Часть 5 | Методы определения SIL (анализ рисков) |
| Часть 6 | Руководство по применению |
| Часть 7 | Методы и средства |

**Жизненный цикл безопасности:**
1. Концепция — определение назначения и предварительный анализ опасностей
2. Анализ опасностей и рисков — идентификация опасностей и оценка рисков
3. Распределение требований безопасности — определение функций безопасности и требуемых уровней SIL
4. Проектирование и реализация
5. Интеграция и испытания
6. Эксплуатация и обслуживание

> *Источник: https://inner.su/articles/funktsionalnaya-bezopasnost-sil-urovni-raschety-pfd-arkhitektury-1oo2-2oo3/*

### 6.2 IEC 61511 / ГОСТ Р МЭК 61511

- **Назначение:** Стандарт функциональной безопасности для **процессных отраслей** (нефтехимия, химия, фармацевтика)
- **Отношение к IEC 61508:** IEC 61511 — отраслевое применение IEC 61508 для непрерывных и пакетных производств
- **Основные понятия:** Приборные функции безопасности (SIF), системы приборной безопасности (SIS)

> *Требует дополнительных источников для детального описания.*

### 6.3 Уровни SIL (1–4)

**SIL (Safety Integrity Level)** — уровень полноты безопасности. Четыре уровня, где SIL 4 — наивысшая надёжность.

| Уровень SIL | PFD (вероятность отказа по запросу) | PFH (вероятность отказа в час) | RRF (фактор снижения риска) | Область применения |
|-------------|-----------------------------------|-------------------------------|----------------------------|-------------------|
| **SIL 4** | ≥ 10⁻⁵ до < 10⁻⁴ | ≥ 10⁻⁹ до < 10⁻⁸ | 10 000 – 100 000 | Критические системы жизнеобеспечения |
| **SIL 3** | ≥ 10⁻⁴ до < 10⁻³ | ≥ 10⁻⁸ до < 10⁻⁷ | 1 000 – 10 000 | Высокорисковые промышленные процессы |
| **SIL 2** | ≥ 10⁻³ до < 10⁻² | ≥ 10⁻⁷ до < 10⁻⁶ | 100 – 1 000 | Стандартные промышленные системы |
| **SIL 1** | ≥ 10⁻² до < 10⁻¹ | ≥ 10⁻⁶ до < 10⁻⁵ | 10 – 100 | Базовые системы с низким риском |

Каждый следующий уровень SIL снижает уровень риска на **порядок (в 10 раз)**.

**Режимы работы:**
- **Низкая интенсивность запросов** — функция безопасности запрашивается не чаще 1 раза в год → используется PFD
- **Высокая интенсивность запросов** — запрашивается чаще 1 раза в год или непрерывно → используется PFH

> *Источник: https://inner.su/articles/funktsionalnaya-bezopasnost-sil-urovni-raschety-pfd-arkhitektury-1oo2-2oo3/*

### 6.4 ПАЗ (Противоаварийная защита)

- **Назначение:** Система, обеспечивающая автоматический перевод технологического объекта в безопасное состояние при возникновении аварийной ситуации
- **Компоненты:** Датчики → Логический решатель (ПЛК безопасности, сертифицированный SIL) → Исполнительные механизмы (отсечные клапаны)
- **Стандарты:** IEC 61511, IEC 61508
- **Принцип:** Независимость от основной системы управления (BPCS)

> *Требует дополнительных источников для детального описания.*

### 6.5 TMR (Тройное модульное резервирование)

**TMR (Triple Modular Redundancy)** — архитектура **2oo3** (два из трёх):
- Три независимых канала выполняют одну и ту же функцию
- Результат определяется **голосованием большинства** (2 из 3)
- Отказ одного канала не влияет на работу системы
- Обеспечивает одновременно **высокую надёжность** и **защиту от ложных срабатываний**
- Максимально достижимый уровень: **SIL 4**

**Формула PFD для 2oo3:**
```
PFDavg = 3 × (λdu × TI)²
```

**Пример:** TMR применяется в системах ПАЗ на опасных химических и нефтеперерабатывающих производствах, АЭС.

> *Источник: https://inner.su/articles/funktsionalnaya-bezopasnost-sil-urovni-raschety-pfd-arkhitektury-1oo2-2oo3/*

### 7.2 РСУ (DCS) vs ПЛК

| Критерий | ПЛК | РСУ (DCS) |
|----------|-----|-----------|
| **Архитектура** | Централизованная или распределённая | Изначально распределённая |
| **Масштаб** | От единиц до тысяч I/O | Тысячи и десятки тысяч I/O |
| **Управление** | Дискретное, логическое, PID | Непрерывное, PID, каскадное, пакетное |
| **Интеграция** | Отдельный контроллер + SCADA | Единая платформа (контроллер + инженерная станция + операторская станция) |
| **Применение** | Конвейеры, упаковка, ЖКХ, малые объекты | Нефтепереработка, химия, энергетика, крупные непрерывные производства |
| **Резервирование** | Опция (за доп. стоимость) | Штатная функция |
| **Стоимость** | Ниже для малых систем | Выше, но дешевле на точку I/O при масштабе |

> *Требует дополнительных источников для детального описания.*

### 7.3 SCADA + ПЛК

**SCADA** (Supervisory Control and Data Acquisition) — система диспетчерского управления и сбора данных. ПЛК интегрируются со SCADA через коммуникационные протоколы (Modbus, OPC UA, Profibus, Ethernet/IP).

**Схема взаимодействия:**
```
[Датчики/Приводы] ↔ [ПЛК] ↔ [Промышленная сеть] ↔ [SCADA-сервер] ↔ [АРМ оператора]
```

- ПЛК **управляет** — выполняет логику, регулирование, блокировки
- SCADA **визуализирует** — показывает мнемосхемы, тренды, журнал событий
- Связь через HMI (панели оператора) или PC-based SCADA

> *Источники: https://www.souz-pribor.ru/articles/kak-rabotaet-plk-kontroller/, https://www.invt.su/articles/programmiruemyy-logicheskiy-kontroller-chto-eto-i-zachem-nuzhen/*

### 7.4 Типовые задачи

| Задача | Описание | Языки |
|--------|----------|-------|
| **Регулирование** | ПИД-регулирование температуры, давления, уровня, расхода. Используются функциональные блоки PID. | FBD, ST, CFC |
| **Блокировки** | Защитные блокировки оборудования: запрет пуска при нарушении условий, аварийный останов | LD, FBD |
| **Сигнализация** | Формирование аварийных и предупредительных сигналов при выходе параметров за границы | LD, ST |
| **Последовательное управление** | Пуск/останов агрегатов по шагам, управление пакетными процессами | SFC |
| **Управление приводами** | Пуск, останов, реверс двигателей; управление частотными преобразователями | LD, ST |
| **Счёт и дозирование** | Подсчёт изделий, дозирование компонентов | ST, FBD |
