# Alpha Licensing Playbook (Internal)

**Назначение:** единый источник истины для расчётов и пресейла по линейке «Альфа-платформа».

## Когда применять

Применять всегда при запросах с ключевыми словами:
- Альфа / Alpha
- лицензирование
- расчёт / пресейл / КП
- SCADA / Platform / One+

Перед ответом:
1. Свериться с этим playbook.
2. Проверить актуальность policy + тарифа.
3. Явно отметить допущения и неизвестные.

1. `policy`

## Жёстко зафиксированные правила

1. Перед финальным расчётом обязательны уточнения:
   - архитектура
   - типы клиентов
   - драйверы/протоколы
   - резервирование
   - terminal/RDP

2. **SCADA с резервом = 2 серверные лицензии.**

3. Перед финальной рекомендацией обязательна сверка:
   - policy
   - тариф

4. В коммерческом предложении (КП):
   - минимум 2 варианта (обычно SCADA + Platform),
   - One+ — для малого/простого сценария.

5. Цены/тарифные детали:
   - не выносить наружу,
   - обсуждать только в прямом контуре со Stanislav.
   - без отдельного разрешения НЕ показывать построчные цены по артикулам; в ответе давать только общую цену системы.

6. Интеграционное правило:
   - обмен One+/SCADA с Platform считать внешним (по умолчанию OPC UA),
   - теги считать внешними у принимающей стороны,
   - исключение «внутренние теги не считаются» — только внутри Platform по Alpha.Link.

7. В расчётах обязателен фокус на драйверы/протоколы
   (OPC/IEC/Modbus/S7 и т.д.).

8. В финальном блоке явно указывать год тарифа.

9. Для Full-клиентов обязательно уточнять terminal/RDP
   (или пометка: «уточняется при заказе ключей»).

## Минимальный чек-лист перед расчётом

1. Состав контуров и серверов (боевые/резервные/DR при наличии)
2. Наличие/тип резервирования по каждому контуру
3. Типы клиентских подключений (толстый/терминальный/WEB)
4. Terminal/RDP (да/нет, объём)
5. Полный список протоколов и драйверов
6. Внешний/внутренний характер каждого обмена
7. Объём тегов/сигналов (по контурам)
8. Требования по Historian/архиву
9. Ограничения по версии policy/тарифного года
10. Какие варианты КП нужны (минимум 2)

## Готовый опросник для клиента (под ТКП)

> Использовать как основную форму сбора данных. Допускается неполное заполнение с явной фиксацией неизвестных.

### 1) Общая рамка
- Заказчик / объект
- Отрасль / тип объекта
- Цель проекта (что автоматизируем)
- Срок запуска / дедлайн ТКП
- Нужен КП в 1 варианте или в 2 вариантах (рекомендуется 2)

### 2) Архитектура
- Планируемое семейство: One+ / SCADA / Platform / не знаю
- Кол-во серверов
- Нужен резерв (да/нет)
- Если да — где именно (SCADA, I/O, Historian и т.д.)
- Площадок/узлов (одна / несколько)

### 3) Данные (теги/сигналы)
- Уже есть оценка внешних тегов? (число)
- Или считать из сигналов (DI/DO/AI/AO)
- DI / DO / AI / AO
- Есть Historian? (да/нет)
- Если да — сколько тегов в историю

### 4) Интеграции и протоколы (обязательно)
- Отметить нужное: OPC UA, OPC DA, Modbus TCP/RTU, IEC 101/104, IEC 61850, S7, BACnet, MQTT, другое
- По каждому источнику:
  - Источник данных
  - Получатель
  - Обмен внутренний (Alpha.Link) или внешний (OPC/другое)

### 5) Клиенты и доступ
- Full-клиенты (кол-во)
- WEB-клиенты одновременные
- Terminal/RDP используется? (да/нет/частично)
- Если частично — распределение по клиентам

### 6) Дополнительные требования
- Отчёты/аналитика
- SLA/поддержка
- Обучение персонала
- Ограничения по инфраструктуре (ОС, виртуализация, сеть, DMZ)
- Особые требования безопасности

### 7) Коммерческие вводные
- Бюджетный коридор (если можно)
- Валюта
- НДС/без НДС
- Нужна поэтапная поставка? (да/нет)
- Нужны альтернативы «бюджет/оптимум/максимум»? (да/нет)

### 8) Неизвестные/на согласовании
- Что пока неизвестно
- Кто со стороны заказчика подтверждает тех.часть
- Ожидаемая дата подтверждения

### Что возвращаем после заполнения
1. Черновик ТКП (структурированный)
2. Вариант A (рекомендованный) + Вариант B (альтернативный)
3. Явный список открытых вопросов
4. Риски и допущения до финализации

---

## Шаблон ответа в пресейле (кратко)

1. Входные параметры (что подтверждено / что предположено)
2. Вариант А (рекомендованный)
3. Вариант Б (альтернативный)
4. Сверка по policy + тарифу (год тарифа)
5. Риски/неизвестные к уточнению
