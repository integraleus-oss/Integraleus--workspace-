# Alpha Licensing Playbook (Internal)

**Назначение:** единый источник истины для расчётов и пресейла по линейке «Альфа-платформа».

## Когда применять

Применять всегда при запросах с ключевыми словами:
- Альфа / Alpha
- лицензирование
- расчёт / пресейл / КП
- SCADA / Platform / One+

Перед ответом:
1. Свериться с этим playbook.
2. Проверить актуальность policy + тарифа.
3. Явно отметить допущения и неизвестные.

---

## Базовые источники (обязательные)

1. `policy`
2. `update-2023`
3. `опросник`
4. `I/O Helper`
5. `тариф / SKU`
6. Рабочая база: `data/licensing_automiq/`
7. Пакет пресейла: `data/licensing_automiq/transfer_package/`

> Если источник недоступен/не найден — не выдумывать, запрашивать у пользователя.

## Жёстко зафиксированные правила

1. Перед финальным расчётом обязательны уточнения:
   - архитектура
   - типы клиентов
   - драйверы/протоколы
   - резервирование
   - terminal/RDP

2. **SCADA с резервом = 2 серверные лицензии.**

3. Перед финальной рекомендацией обязательна сверка:
   - policy
   - тариф

4. В коммерческом предложении (КП):
   - минимум 2 варианта (обычно SCADA + Platform),
   - One+ — для малого/простого сценария.

5. Цены/тарифные детали:
   - не выносить наружу,
   - обсуждать только в прямом контуре со Stanislav.

6. Интеграционное правило:
   - обмен One+/SCADA с Platform считать внешним (по умолчанию OPC UA),
   - теги считать внешними у принимающей стороны,
   - исключение «внутренние теги не считаются» — только внутри Platform по Alpha.Link.

7. В расчётах обязателен фокус на драйверы/протоколы
   (OPC/IEC/Modbus/S7 и т.д.).

8. В финальном блоке явно указывать год тарифа.

9. Для Full-клиентов обязательно уточнять terminal/RDP
   (или пометка: «уточняется при заказе ключей»).

## Кейсовые ориентиры (исторические)

- 5700 внешних тегов → уровень `SCADA-SRV15k`
- Historian 1710 → `HIST-S2k`

> Использовать как ориентир, не как абсолют без сверки policy/тарифа.

---

## Минимальный чек-лист перед расчётом

1. Состав контуров и серверов (prod/test/dr/reserve)
2. Наличие/тип резервирования по каждому контуру
3. Типы клиентских подключений (толстый/терминальный/WEB)
4. Terminal/RDP (да/нет, объём)
5. Полный список протоколов и драйверов
6. Внешний/внутренний характер каждого обмена
7. Объём тегов/сигналов (по контурам)
8. Требования по Historian/архиву
9. Ограничения по версии policy/тарифного года
10. Какие варианты КП нужны (минимум 2)

---

## Шаблон ответа в пресейле (кратко)

1. Входные параметры (что подтверждено / что предположено)
2. Вариант А (рекомендованный)
3. Вариант Б (альтернативный)
4. Сверка по policy + тарифу (год тарифа)
5. Риски/неизвестные к уточнению
