# MATRIX_ARCH.md — Границы архитектур и выбор семейства

Источник истины для пресейла по выбору One+ / SCADA / Platform.

## 1) Базовая матрица выбора

### Alpha.One+
Выбирать, когда:
- небольшой/простой сценарий;
- однопользовательский формат;
- без резервирования и без дополнительных клиентов.

Ограничения:
- One+ — однопользовательский комплект.
- Допускается 1 WEB-клиент (по policy).
- Масштабируемые WEB-клиенты (CL-RO и т.п.) не для One+.

### Alpha.SCADA
Выбирать, когда:
- типовой промышленный контур;
- нужна схема основной + резервный сервер;
- количество серверов системы не выходит за рамки SCADA-подхода.

Правила:
- при резерве серверная лицензия дублируется на тот же уровень;
- при >5 WEB-подключениях добавлять рекомендацию выделенного WEB-сервера;
- при выделенном WEB-сервере использовать WEB-PORTAL.

### Alpha.Platform
Выбирать, когда:
- нужна мультисерверная архитектура/масштабирование;
- ожидается рост системы и расширение контуров;
- требуется архитектурная гибкость на будущее.

Правила:
- лицензируется по тегам системы, числу физических I/O серверов и клиентам;
- в Platform драйверы входят в состав, отдельно не докупаются;
- внутренние теги не считаются внешними только при Alpha.Link.

## 2) Жёсткие переключатели

1. Если мультисерверный сценарий явно выходит за SCADA-пределы — предлагать Platform.
2. Если в требованиях есть резервирование — сразу базовый вариант Alpha.SCADA (резерв = 2 серверные лицензии одного уровня).
3. Если резервирования нет и кейс до 50 000 тегов при 1 клиенте — допустимо предлагать Alpha.One+ как вариант.
4. One+ в остальных случаях предлагать только для малого односерверного сценария без расширенной клиентской модели.
5. Всегда сверять финал по policy + тарифу актуального года.

## 3) Вывод в ТКП

- По умолчанию: 2 варианта (SCADA + Platform), если не однотипный малый кейс.
- Для Platform обязательно писать: драйверы включены.
- Для цен: без отдельного разрешения не показывать построчные цены по SKU.
