ТЕХНИЧЕСКОЕ ЗАДАНИЕ
на разработку и внедрение SCADA-системы
в составе открытой архитектуры АСУ ТП на основе O-PAS Standard Version 2.1
Версия 7 — Полное описание функционала

1. ОБЩИЕ ПОЛОЖЕНИЯ

1.1. Назначение документа

Настоящее техническое задание определяет полный состав функциональных и нефункциональных требований к SCADA-системе, разрабатываемой для применения в составе открытой архитектуры автоматизированной системы управления технологическими процессами на основе O-PAS Standard Version 2.1.

Документ предназначен для использования:
- разработчиком SCADA-системы — для проектирования и реализации решения;
- архитектором АСУ ТП — для согласования архитектурных границ;
- заказчиком — для приемки и верификации функционала;
- интегратором — для планирования интеграционных работ;
- службой ИБ — для оценки соответствия требованиям безопасности.

1.2. Область применения

SCADA-система предназначена для промышленных объектов непрерывного и дискретного производства, включая нефтехимию, нефтегаз, энергетику, металлургию, водоподготовку, фармацевтику и смежные отрасли. Система должна быть пригодна для эксплуатации на объектах с высокими требованиями к доступности, масштабируемости, кибербезопасности, трассируемости и поэтапной модернизации.

1.3. Ключевые принципы

Разрабатываемая SCADA-система должна соответствовать следующим принципам:
- открытая архитектура без необоснованного vendor lock-in;
- supervisory-уровень: система не подменяет функции базового управления и safety;
- переносимая конфигурация в документированных форматах;
- интероперабельность через стандартизованные интерфейсы;
- встроенная безопасность на уровне не ниже SL2 по IEC 62443-4-2;
- сопровождаемость проекта другим квалифицированным исполнителем;
- модульность с возможностью поэтапного расширения.

3. АРХИТЕКТУРНЫЙ КОНТЕКСТ

3.1. Место SCADA в архитектуре O-PAS

SCADA-система занимает supervisory-уровень и взаимодействует с распределенными узлами управления, шлюзами, сервисами и инженерными инструментами через O-PAS Connectivity Framework. Система не является:
- платформой исполнения control applications реального времени;
- заменой SIS, ESD, F&G и иных safety-подсистем;
- единственным источником технологической истины для нижестоящих контуров управления;
- закрытой engineering-средой, препятствующей дальнейшему сопровождению.

3.2. Модель приложений FILO

O-PAS определяет четырехуровневую модель:
- Layer-F — портируемые приложения (function blocks, IEC 61131-3). Intrinsically portable, без перекомпиляции;
- Layer-I — intermediate (execution engines);
- Layer-L — library/services (OPC UA, database, tag services);
- Layer-O — операционная система.

SCADA взаимодействует с Layer-F через OCF и использует сервисы Layer-L.

3.3. Типы распределенных узлов

SCADA должна интегрироваться со всеми типами DCN:
- DCN с физическим I/O — прямой сбор полевых данных;
- DCN без I/O — вычислительные, координирующие;
- Virtual DCN — виртуализованные узлы;
- ACP (Advanced Computing Platform) — для ресурсоемких задач (MPC, аналитика, historian);
- Gateway — обеспечение OCF-совместимости legacy-систем (DCS, PLC, HART, Foundation Fieldbus, Modbus, PROFIBUS).

При интеграции через Gateway SCADA должна учитывать возможную потерю части метаданных legacy-протоколов и корректно обрабатывать ограниченный набор quality/status/timestamp.

3.4. Quality attributes

При проектировании должны учитываться: Interoperability, Portability, Modularity, Interchangeability, Scalability, Maintainability, Testability, Traceability, Resilience, Usability, Securability, Availability, Configurability, Evolvability.

5. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

5.1. ПОДСИСТЕМА ВИЗУАЛИЗАЦИИ

5.1.1. Типы экранов

SCADA-система должна обеспечивать создание и эксплуатацию следующих видов экранов:
- общесистемный overview — обзор всей площадки или завода;
- area overview — обзор технологической зоны или участка;
- unit overview — обзор технологической установки;
- equipment faceplate — детальный экран единицы оборудования с параметрами, режимами, командами;
- P&ID-style мнемосхемы — технологические схемы с привязкой к реальным параметрам;
- экран контура регулирования — PID faceplate с отображением PV, SP, OUT, режим, ошибки, тренд;
- экран группы контуров — сводное представление нескольких контуров;
- экран состояния последовательности (sequence display) — для batch/sequence management, если применимо;
- экран тревог и событий — сводный и по объектам;
- экран трендов — online и historical;
- экран навигации — карта экранов, поиск объектов;
- диагностический экран — состояние связности, серверов, архивов, времени;
- инженерный экран — конфигурация, версии, аудит;
- административный экран — управление ролями, пользователями, сессиями.

5.1.2. Содержание и представление данных

Для каждого отображаемого параметра должны быть доступны:
- наименование (tag name / display name);
- текущее значение;
- единица измерения;
- диапазон шкалы;
- временная метка;
- статус качества (good / bad / uncertain / stale);
- тревожный статус (нормальный / активная тревога / неподтвержденная / подавленная);
- режим работы (auto / manual / cascade / remote cascade / out of service), если применимо;
- источник данных;
- признак simulation, если активен.

5.1.3. Элементы управления на экранах

SCADA должна обеспечивать размещение на экранах следующих интерактивных элементов:
- индикаторы аналоговых значений (шкалы, столбики, числа);
- индикаторы дискретных состояний (ON/OFF, открыт/закрыт, работает/стоит);
- кнопки управления (пуск, стоп, открыть, закрыть, квитировать);
- поля ввода уставок и параметров;
- переключатели режимов;
- индикаторы тревог (цветовые, мигающие, текстовые);
- minitrends (встроенные тренды на faceplate);
- таблицы параметров;
- ссылки навигации;
- контекстные меню объектов.

5.1.4. Навигация

Система должна обеспечивать:
- иерархическую навигацию: площадка → зона → установка → оборудование → параметр;
- поиск объектов по имени, тегу, описанию;
- навигацию от тревоги к источнику;
- навигацию от тренда к объекту;
- возврат к предыдущему экрану;
- закладки и избранное, если применимо;
- карту экранов.

5.1.5. Визуальные правила

Система должна поддерживать:
- цветовые схемы, согласованные с эксплуатационными стандартами объекта;
- различение нормального, тревожного, диагностического, сервисного и отказного состояний;
- мигание для неподтвержденных тревог;
- затемнение или специальную индикацию для данных с плохим качеством;
- запрет отображения невалидных данных как достоверных;
- поддержку темной и светлой тем, если это требуется эксплуатацией.

5.2. ПОДСИСТЕМА ДИСПЕТЧЕРСКОГО УПРАВЛЕНИЯ

5.2.1. Операторские команды

SCADA должна обеспечивать выполнение следующих типов команд:
- изменение уставки (setpoint change);
- изменение выходного значения (output override);
- переключение режима контура (auto/manual/cascade/remote cascade);
- пуск и останов оборудования;
- открытие и закрытие запорной арматуры;
- подтверждение (acknowledge) тревог;
- квитирование (reset) тревог;
- shelving (временное подавление) тревог;
- suppress (подавление) тревог;
- перевод объекта в состояние out of service;
- возврат объекта в работу;
- принудительное задание значения (force/override), если разрешено проектом;
- снятие принудительного значения;
- запуск и останов supervisory-процедур, если применимо.

5.2.2. Безопасность команд

Для каждой команды система должна обеспечивать:
- проверку полномочий пользователя до выполнения;
- подтверждение оператором перед выполнением критических команд;
- двухэтапное подтверждение для особо опасных действий (определяется проектом);
- регистрацию в аудиторском журнале: кто, когда, что, на каком объекте, с каким результатом;
- отображение результата выполнения команды (успех / отказ / timeout);
- блокировку команды при отсутствии связи с объектом управления;
- блокировку команды при состоянии out of service, если это предусмотрено политикой.

5.2.3. Защита от ошибочных действий

Система должна обеспечивать:
- проверку диапазона при вводе уставок (range checking);
- предупреждение при вводе значений за пределами нормального диапазона;
- невозможность отправки команды с некорректными параметрами;
- визуальное отличие опасных команд от штатных;
- подтверждение перед отправкой команд на несколько объектов одновременно.

5.3. ПОДСИСТЕМА ПРИЕМА И ОБРАБОТКИ ДАННЫХ

5.3.1. Источники данных

SCADA должна принимать данные от:
- DCN с физическим I/O;
- DCN без I/O;
- Virtual DCN;
- ACP;
- Gateway (legacy DCS, PLC, HART, FF, Modbus, PROFIBUS);
- PLC и RTU, подключенные через OPC UA;
- внешние historian-серверы;
- внешние аналитические и диагностические системы;
- manual data entry (ручной ввод лабораторных данных), если предусмотрено проектом.

5.3.2. Типы данных

Система должна корректно обрабатывать:
- аналоговые сигналы (temperature, pressure, flow, level и т.д.);
- дискретные сигналы (on/off, open/close, run/stop);
- многосостояния (multi-state: режимы работы, позиции);
- расчетные параметры (derived / calculated values);
- накопительные значения (totalizers, counters);
- диагностические признаки (device health, communication status);
- строковые значения (batch ID, recipe name, operator comment);
- временные метки событий;
- trigger-сигналы.

5.3.3. Допустимые O-PAS signal types

Система должна поддерживать только допустимые O-PAS типы сигналов:
- AnalogUnitRangeType;
- TwoStateDiscreteType;
- MultiStateValueDiscreteType;
- MultiStateDictionaryEntryDiscreteBaseType;
- OPASAnalogSignalType;
- TriggerType;
- PA-DIM типы: AnalogSignalVariableType, DiscreteSignalVariableType, TwoStateDiscreteSignalVariableType, MultiStateDiscreteSignalVariableType;
- или типы, производные от вышеперечисленных.

5.3.4. Метаданные

Для каждого принимаемого сигнала должны обрабатываться:
- source identifier;
- timestamp (предпочтительно от источника);
- quality/status (good / bad / uncertain / stale);
- engineering units;
- диапазон (range: low/high scale);
- alarm relevance;
- объектная принадлежность;
- описание / комментарий.

5.3.5. Обработка аномалий

Система должна корректно обрабатывать:
- потерю связи с источником;
- получение данных с bad quality;
- устаревание данных (stale detection);
- переполнение буферов приема;
- рассинхронизацию времени с источником;
- получение данных через gateway с ограниченным набором метаданных;
- обнаружение дублирующихся или конфликтующих значений.

5.3.6. Расчетные параметры

SCADA должна поддерживать:
- создание расчетных переменных на основе нескольких входных сигналов;
- базовые арифметические и логические операции;
- условные выражения;
- агрегатные функции (min, max, avg, sum за интервал);
- totalizing (накопление);
- dead-band обработку для подавления шума;
- rate of change;
- задержки (delay, lead/lag), если требуется проектом.

5.4. ПОДСИСТЕМА ТРЕВОГ И СОБЫТИЙ

5.4.1. Общие требования

Подсистема должна быть совместима с моделью O-PAS Part 6.3, OPC UA Part 9 Alarms and Conditions и учитывать рекомендации ISA-18.2 и EEMUA 191.

5.4.2. Иерархия типов тревог

Система должна поддерживать:
- LimitAlarm: ExclusiveLevel, NonExclusiveLevel, ExclusiveDeviation, NonExclusiveDeviation, ExclusiveRateOfChange, NonExclusiveRateOfChange;
- DiscreteAlarm: OffNormalAlarm, TripAlarm, InstrumentDiagnosticAlarm, SystemDiagnosticAlarm, CertificateExpirationAlarm;
- DiscrepancyAlarm.

5.4.3. Exclusive и Non-Exclusive

Exclusive — показывает только одну тревогу данного типа (High ИЛИ High-High).
Non-Exclusive — показывает все активные одновременно (и High, и High-High).

5.4.4. Пороговые тревоги

Для аналоговых сигналов должны поддерживаться:
- High-High Limit (HH);
- High Limit (H);
- Low Limit (L);
- Low-Low Limit (LL);
- базовые значения порогов (BaseHighHighLimit, BaseHighLimit, BaseLowLimit, BaseLowLowLimit);
- Deviation High / Deviation Low (отклонение от уставки);
- Rate of Change (скорость изменения).

5.4.5. Дискретные тревоги

Для дискретных сигналов должны поддерживаться:
- OffNormal — отклонение от нормального состояния;
- Trip — срабатывание защиты;
- State alarm — тревога по состоянию;
- Diagnostic alarm — диагностическая.

5.4.6. Атрибуты тревог

Для каждой тревоги должны поддерживаться:
- Severity (приоритет, 0-1000 по OPC UA);
- ConditionName — наименование условия;
- ConditionClassId / ConditionClassName — класс тревоги;
- ConditionSubClassId / ConditionSubClassName — подкласс;
- SourceNode / SourceName — объект-источник;
- Message — текст сообщения;
- InputNode — привязка к входному сигналу;
- MaxTimeShelved — максимальное время shelving;
- OnDelay — задержка срабатывания;
- OffDelay — задержка возврата в норму;
- ReAlarmTime — время повторного оповещения;
- ReAlarmRepeatCount — количество повторов;
- SilenceState — состояние звука;
- EnabledState — включена/выключена.

5.4.7. Состояния тревоги

Система должна поддерживать следующие состояния:
- ActiveState — активна / не активна;
- AckedState — подтверждена / не подтверждена;
- ConfirmedState — подтверждение второго уровня, если применимо;
- SuppressedState — подавлена;
- OutOfServiceState — выведена из обслуживания;
- ShelvedState — временно скрыта (TimedShelved / OneShotShelved / Unshelved);
- LatchedState — защелкнута;
- SuppressedOrShelved — агрегированный признак подавления.

5.4.8. Жизненный цикл тревоги

Система должна регистрировать и отображать полный lifecycle:
- возникновение (raise);
- активация (active);
- уведомление оператора;
- подтверждение (acknowledge);
- подтверждение второго уровня (confirm), если применимо;
- возврат в норму (return to normal);
- сброс / очистка;
- повторное срабатывание (re-alarm), если настроено.

5.4.9. Управление тревогами

Система должна поддерживать:
- acknowledge (подтверждение);
- shelving (временное подавление) с ограничением MaxTimeShelved;
- suppression (подавление);
- out of service (вывод из обслуживания);
- enable/disable;
- silence (отключение звука);
- группировку тревог (alarm groups);
- фильтрацию по приоритету, источнику, типу, состоянию, времени;
- сортировку;
- поиск;
- навигацию к источнику тревоги;
- навигацию к связанному экрану;
- контекстное меню тревоги.

5.4.10. Экран тревог

Экран активных тревог должен отображать:
- timestamp возникновения;
- timestamp подтверждения;
- источник (tag / equipment);
- текст сообщения;
- приоритет;
- состояние (active, acked, return-to-normal, shelved и т.д.);
- текущее значение параметра, вызвавшего тревогу;
- имя оператора, подтвердившего тревогу.

5.4.11. Alarm summary и KPI

Система должна поддерживать:
- alarm summary по приоритетам;
- alarm summary по объектам/установкам;
- standing alarm list;
- alarm rate (тревог в час);
- stale alarm detection;
- chattering alarm detection;
- flood alarm detection;
- базовую alarm performance analytics по ISA-18.2 / EEMUA 191.

5.4.12. Экспорт/импорт

Система должна поддерживать экспорт и импорт конфигурации тревог в документированных форматах.

5.5. ПОДСИСТЕМА ЖУРНАЛИРОВАНИЯ И АУДИТА

5.5.1. Типы журналов

SCADA должна вести:
- Event Journal — хронологический журнал всех событий;
- Alarm Journal — журнал тревог с полным lifecycle;
- Operator Action Log — журнал действий операторов;
- Engineering Change Log — журнал инженерных изменений;
- Security Event Log — журнал security-событий;
- System Diagnostic Log — журнал системной диагностики;
- Communication Log — журнал состояния связности с источниками;
- Login/Logout Log — журнал входов и выходов пользователей.

5.5.2. Атрибуты записи

Каждая запись должна содержать:
- уникальный идентификатор;
- timestamp (UTC и локальное время);
- тип события;
- источник;
- инициатор (пользователь / система / устройство);
- описание / текст;
- старое и новое значение, если применимо;
- результат действия;
- приоритет, если применимо.

5.5.3. Функции журналов

Должны поддерживаться:
- просмотр в хронологическом порядке;
- фильтрация по времени, типу, источнику, пользователю;
- полнотекстовый поиск;
- экспорт выборки;
- печать;
- архивирование и ротация;
- политика хранения (retention policy).

5.5.4. Защита журналов

Журналы должны быть защищены от:
- несанкционированного изменения;
- несанкционированного удаления;
- переполнения без уведомления.

5.6. ПОДСИСТЕМА ТРЕНДОВ И ИСТОРИЗАЦИИ

5.6.1. Типы трендов

Система должна обеспечивать:
- online trends — отображение текущих значений в реальном времени;
- historical trends — отображение архивных данных за произвольный период;
- combined trends — наложение online и historical;
- multi-pen trends — одновременное отображение нескольких параметров (не менее 8);
- XY plots — отображение зависимости одного параметра от другого;
- bar charts — столбчатые диаграммы для сравнения;
- tabular view — табличный просмотр значений.

5.6.2. Функции трендов

Должны поддерживаться:
- масштабирование по времени (zoom in/out);
- масштабирование по значению;
- прокрутка по времени (pan);
- курсор с отображением точного значения, времени и качества;
- annotations — маркеры событий и тревог на временной шкале;
- экспорт данных тренда;
- сохранение конфигурации тренда (template);
- печать тренда.

5.6.3. Архивирование значений

Система должна обеспечивать:
- непрерывное архивирование с заданным интервалом;
- архивирование по изменению (exception-based / deadband);
- архивирование по событию;
- хранение: timestamp, value, quality/status, source;
- агрегацию: min, max, avg, count, sum, range за интервал;
- сжатие архивных данных;
- политику хранения (retention) по классам данных;
- экспорт архивных данных.

5.6.4. Производительность архива

Архивирование не должно ухудшать online-функции визуализации, тревог и управления.

5.7. ПОДСИСТЕМА ИНЖЕНЕРНОГО КОНФИГУРИРОВАНИЯ

5.7.1. Объекты конфигурирования

SCADA должна обеспечивать создание и редактирование:
- иерархии объектов (площадка / зона / установка / оборудование / параметр);
- тегов и привязки к источникам данных;
- экранных форм и мнемосхем;
- графических шаблонов и библиотек символов;
- конфигурации тревог (типы, пороги, приоритеты, задержки);
- конфигурации архивирования (какие параметры, интервалы, retention);
- конфигурации отчетов;
- конфигурации ролей и прав доступа;
- интеграционных соединений и привязок;
- расчетных параметров и формул.

5.7.2. Версионирование и контроль изменений

Система должна поддерживать:
- контроль версий конфигурации;
- сравнение версий (diff);
- откат к ранее подтвержденной версии;
- маркировку версий (release tagging);
- журналирование всех изменений с указанием автора, времени и описания;
- процедуру согласования изменений, если требуется проектом.

5.7.3. Online и offline конфигурирование

Система должна поддерживать:
- offline конфигурирование — разработка проекта без подключения к промышленной среде;
- online конфигурирование — внесение изменений в работающую систему с контролем влияния;
- partial download — загрузка только измененной части конфигурации;
- pre-deployment validation — проверка конфигурации перед применением.

5.7.4. Шаблоны и библиотеки

Система должна поддерживать:
- шаблоны объектов (equipment templates);
- библиотеки графических символов;
- библиотеки экранных форм;
- библиотеки конфигураций тревог;
- наследование от шаблона;
- массовое создание объектов по шаблону.

5.8. ПОДСИСТЕМА ОТЧЕТНОСТИ

5.8.1. Типы отчетов

SCADA должна поддерживать:
- суточные отчеты;
- сменные отчеты;
- отчеты по тревогам и аварийности;
- отчеты по действиям операторов;
- отчеты по состоянию оборудования;
- отчеты по доступности каналов связи;
- отчеты по инженерным изменениям;
- отчеты по security-событиям;
- alarm KPI report (ISA-18.2 metrics);
- произвольные отчеты по выборке.

5.8.2. Функции отчетов

Должны поддерживаться:
- автоматическое формирование по расписанию;
- формирование по запросу;
- параметризация (период, объект, тип);
- экспорт в PDF, Excel, CSV;
- печать;
- рассылка по email / сохранение в сетевую папку, если предусмотрено проектом.

5.9. ПОДСИСТЕМА ДИАГНОСТИКИ И ЭКСПЛУАТАЦИОННОГО КОНТРОЛЯ

5.9.1. Контролируемые параметры

Система должна обеспечивать контроль:
- состояния серверных ролей (primary, standby, failed);
- состояния соединений с каждым источником данных;
- состояния архивных подсистем;
- состояния подсистемы тревог;
- состояния синхронизации времени;
- загрузки CPU, памяти, дисков;
- количества активных сессий;
- скорости поступления данных (tag update rate);
- задержки обработки данных (latency);
- состояния лицензий, если применимо;
- диагностических сообщений от приложений.

5.9.2. Диагностические экраны

Должны быть предусмотрены:
- системный overview — сводное состояние всех подсистем;
- экран коммуникаций — состояние каждого канала связи;
- экран серверов — состояние серверных процессов;
- экран архивов — объем, заполнение, скорость записи;
- экран времени — источники, drift, status;
- экран сессий — активные пользователи, роли, действия.

5.9.3. Автоматическая диагностика

Система должна:
- генерировать системные тревоги при обнаружении проблем;
- регистрировать диагностические события в журнале;
- поддерживать health check и self-test;
- обеспечивать уведомление эксплуатационного персонала о деградации.

5.10. ПОДСИСТЕМА УПРАВЛЕНИЯ ДОСТУПОМ

5.10.1. Роли

Должны поддерживаться как минимум:
- Оператор (уровень 1) — мониторинг и базовые команды;
- Старший оператор (уровень 2) — расширенные команды, shelving тревог;
- Инженер КИПиА — инженерное конфигурирование, диагностика;
- Инженер АСУ — полное конфигурирование системы;
- Администратор — управление ролями, пользователями, политиками;
- Сервисный инженер — техническое обслуживание, обновления;
- Аудитор — read-only доступ к журналам и конфигурации;
- Гость — ограниченный мониторинг, без команд.

5.10.2. Разграничение

Система должна обеспечивать:
- привязку полномочий к ролям;
- привязку ролей к пользователям;
- разграничение доступа к экранам, командам, объектам, конфигурации;
- разграничение по зонам/установкам (area-based access), если требуется;
- наследование полномочий;
- временное повышение полномочий (elevation), если предусмотрено.

5.11. ПОДСИСТЕМА РЕЗЕРВНОГО КОПИРОВАНИЯ И ВОССТАНОВЛЕНИЯ

5.11.1. Объекты резервирования

Должны резервироваться:
- проектная конфигурация;
- настройки тревог;
- настройки архивирования;
- экранные формы;
- роли и права;
- журналы и архивы;
- сертификаты;
- системные настройки.

5.11.2. Функции

Система должна обеспечивать:
- ручное и автоматическое резервное копирование;
- проверку целостности резервных копий;
- восстановление из резервной копии;
- выборочное восстановление (отдельный компонент);
- документирование процедур backup/restore.

6. ТРЕБОВАНИЯ К ИНТЕГРАЦИИ И СВЯЗНОСТИ

6.1. OCF-профили

6.1.1. Обязательно: OCF-001 (OPC UA Client/Server Profile).
6.1.2. Где применимо: OCF-002 (OPC UA Client/Server and PubSub).
6.1.3. Обязательно: OCF-F-003 (OCF Client Facet), OCF-F-005 (Alarms and Conditions Client).
6.1.4. Если SCADA — сервер: OCF-F-001 (Security Server), OCF-F-004 (A&C Server).

6.2. OPC UA Security

6.2.1. SecurityPolicy: Aes128-Sha256-RsaOaep и Basic256Sha256.
6.2.2. User Token JWT.
6.2.3. Global Certificate Management.
6.2.4. AliasName support.

6.3. Discovery и namespace

6.3.1. GDS-001 (Global Discovery Server), где развернут.
6.3.2. Корректная обработка OPC UA namespaces, включая O-PAS CF namespace.

6.4. Синхронизация времени

6.4.1. NET-F-011 (NTP/OS-based) — обязательно.
6.4.2. NET-F-012 / NET-F-013 (PTP) — где применимо.

6.5. Интеграция со всеми типами DCN, PLC, RTU, historian, analytics, legacy gateways.

7. ТРЕБОВАНИЯ К МОДЕЛИ ДАННЫХ

7.1. Иерархическая модель: площадка → зона → установка → оборудование → параметр. Иерархии ISA-88, ISA-95.

7.2. Атрибуты DCN: Manufacturer, Model, SerialNumber, SoftwareRevision, HardwareRevision, DeviceHealth, AssetId, ComponentName.

7.3. Function block model: Inputs, Outputs, Configuration, Tuning, Status, Operational. FBErrorType: OutOfService, ConfigurationError, SimulationActive, PvHasBadStatus.

7.4. Единые правила именования для всех сущностей.

8. ТРЕБОВАНИЯ К ИНЖЕНЕРНОМУ ОБМЕНУ

8.1. Import/export в документированных форматах (AML, OPC UA NodeSet, Open Packaging).
8.2. Не закрытый формат как единственный носитель.
8.3. Валидация, контроль целостности, сравнение, откат.
8.4. Переносимость Layer-F конфигурации.

9. ТРЕБОВАНИЯ К КАЧЕСТВУ ДАННЫХ И ВРЕМЕНИ

9.1. Различение: Good, Bad, Uncertain, Stale.
9.2. Невалидные данные не отображаются как достоверные.
9.3. Timestamp источника при его наличии.
9.4. Нарушение синхронизации — системное событие.
9.5. Поддержка NAMUR NE107 диагностических состояний: Failure, Function Check, Out of Specification, Maintenance Required.

10. ТРЕБОВАНИЯ К ИНФОРМАЦИОННОЙ БЕЗОПАСНОСТИ

10.1. Общий уровень: SL2 по IEC 62443-4-2 как минимум.

10.2. FR1 — Идентификация и аутентификация:
- уникальная идентификация пользователей;
- аутентификация процессов и устройств;
- управление учетными записями;
- стойкость паролей;
- PKI / X.509;
- реагирование на неудачные попытки входа;
- system use notification.

10.3. FR2 — Управление использованием:
- RBAC;
- session lock;
- remote session termination;
- auditable events;
- timestamps и time sync для аудита;
- non-repudiation.

10.4. FR3 — Целостность системы:
- communication integrity и authentication;
- защита от вредоносного кода;
- верификация security-функций;
- контроль целостности ПО;
- input validation;
- deterministic output;
- защита аудита;
- поддержка обновлений с проверкой authenticity.

10.5. FR4 — Конфиденциальность:
- защита конфиденциальной информации;
- очистка при удалении;
- криптография по политике.

10.6. FR5 — Ограничение потоков: сетевая сегментация.

10.7. FR6 — Реагирование: доступность аудита, continuous monitoring.

10.8. FR7 — Доступность ресурсов:
- DoS protection;
- resource management;
- backup с проверкой целостности;
- recovery;
- least functionality;
- инвентаризация компонентов.

11. ТРЕБОВАНИЯ К НАДЕЖНОСТИ И ДЕГРАДАЦИИ

11.1. Резервирование критических серверных ролей.
11.2. Hot standby для серверов визуализации и тревог, если предусмотрено проектом.
11.3. Описание поведения при отказах каждой подсистемы.
11.4. Предсказуемый режим деградации.
11.5. Сохранение индикации деградации для оператора.
11.6. Независимость нижестоящего управления от SCADA.
11.7. Автоматическое восстановление после transient failures.
11.8. Корректное поведение при cold/warm/hot restart.

12. ТРЕБОВАНИЯ К ПРОИЗВОДИТЕЛЬНОСТИ

12.1. Время обновления экрана: не более 2 секунд для критических экранов при нормальной нагрузке.
12.2. Время реакции на команду оператора: не более 2 секунд до отображения результата.
12.3. Время появления тревоги на экране: не более 2 секунд от момента получения.
12.4. Throughput архивирования: определяется проектом, подтверждается испытаниями.
12.5. Максимальное количество тегов, пользователей, соединений: определяется проектом.
12.6. Производительность должна подтверждаться нагрузочными испытаниями.

13. ТРЕБОВАНИЯ К РАЗРАБОТКЕ И СОПРОВОЖДЕНИЮ

13.1. Архитектурное описание решения.
13.2. Документированная проектная структура.
13.3. Все интерфейсы, ограничения и нестандартные решения документированы.

14.1. Программа и методика испытаний.
14.2. Обязательные испытания: документация, модель данных, экраны, навигация, тревоги, архивы, тренды, отчеты, роли, аудит, import/export, backup/restore, интеграция, деградация, производительность, FAT, SAT.

17. ОЖИДАЕМЫЙ РЕЗУЛЬТАТ

Результатом должна являться внедренная и документированная SCADA-система, совместимая с O-PAS v2.1, обеспечивающая полный набор supervisory-функций: визуализацию, диспетчерское управление, развитую подсистему тревог и событий, архивирование, аудит, отчетность, диагностику, инженерное конфигурирование, защищенную эксплуатацию на уровне SL2 по IEC 62443, и пригодная для масштабирования, сопровождения и дальнейшего развития без vendor lock-in.
